# VALUE_FOR_REPOMIND.md — 给 RepoMind 团队的价值主张(产品决策语言)

> 这份文档不堆实验细节,只回答一个问题:**这套研究对你们的产品、资源分配、技术路线意味着什么。**
> 关键结论各配一个最硬的数据支撑;完整证据见 `INSIGHT.md` / `EXPERIMENTS.md`。

---

## 一句话结论
**对"代码跨文件依赖理解"这类任务,把工程资源投在「依赖结构的静态分析质量+覆盖」和「检索深度」上,
回报是主导级的大效应(严打口径约 4-5×、宽松匹配口径约 2×;〔M56 降级〕不再称"数量级",见下注);
投在「更大底座模型 / 更复杂 prompt / 微调 / 完整图数据库 RAG」上,经穷尽验证回报都是边际(统计噪声内),有的还原理上不可能超越。**

> ⚠ **M56 措辞降级(第 11 次诚实修正)**:① "数量级"→"主导杠杆/大效应"(严打 4-5×、宽松 2×,近零 baseline 部分是精确召回严打产物);② "judge-independent"→"LLM-judge-independent(parser-anchored)"。本文一切处按此读。

## 1. 资源优先级(按实测杠杆排序)

| 投入方向 | 实测效应 | 决策含义 |
|---|---|---|
| **依赖结构静态分析(算准算全)** | baseline 0.05→0.94,**+0.5~0.9 数量级杠杆**(p<1e-18,跨 Go/Java/C/CPython) | **第一优先级,钱花这里** |
| **检索深度/层级(找到对的卡片)** | rerank 12→80 **+0.14**;工业级 1456 池仍捕获 97.5% | 第二优先级,大项目尤其重要 |
| 多跳/间接依赖:预先物化传递路径 | 间接依赖 0.40→**0.99**(B2 物化) | 针对性投入,见下 §2 |
| ——————分割线:以下都是边际,别多投—————— | | |
| 更大底座模型 / 代码专精模型 | 通用 14B ≥ Coder 14B(本任务) | 不必为此任务追更大/更专模型 |
| Prompt 工程(System/CoT/few-shot) | 经独立裁判**全部不显著**(CoT 原 +0.02 是污染伪影,已收回) | **不值得投精力** |
| 微调(生成器/检索器/PEFT/DAPT/RL) | 生成器 +0.04 冗余、检索器 −0.02、DAPT 无、**RL 检索器 0.958<静态 1.000** | **对"注入依赖知识"投入产出极低** |
| 完整图数据库 RAG(图结构子图检索) | B3 子图检索 0.69 **<** 轻量文本化物化 0.90 | **过度工程,不建议上** |

**为什么模型侧一律边际**:依赖关系是"**算**"出来的结构事实(A import B 是客观的),不是"**读/猜**"出来的统计规律。
所以能把它算出来的静态分析是天花板;让模型去读、去猜、去学着逼近,都赢不过直接算。

### 1b. 私有代码迁移性(M42 实测,甲方最关心的"会不会缩水")
你们跑**私有代码**,模型没背过——那"+0.9 碾压"会不会有一大块是模型**背过开源库**帮的忙、到私有代码就缩水?**实测答案:不会。**
把 werkzeug 标识符**全改名**(模拟"模型没见过这套代码"),结构化卡片性能 **100% 保持**(seen 98.6%→unseen 100%,无差异);而模型**纯靠记忆**的依赖召回只有 ~15%(改名后掉到 ~2%)。
→ **结构化卡片的价值 ~100% 来自"把结构喂进上下文",~0 来自"模型背过你的代码";预期干净迁移到 RepoMind 私有代码,不缩水。**
**边界(诚实)**:改名只灭"表层标识符记忆",真实私有代码的结构习惯可能不同;且这条说的是"卡片喂进去模型能用",**不**保证"卡片本身在私有代码上能被静态分析算准算全"——后者仍取决于静态分析覆盖率(见 §3 边界)。详见 `MEMORY_VS_STRUCTURE.md`。

## 2. 给 RepoMind 的具体可执行建议(基于实证)

1. **把人力投在"依赖结构静态分析做准做全"**,尤其覆盖:
   - **别名/re-export 解析**(如 `import X as _Y` 后 `class A(_Y)`)——实测攻克后符号失联类用例 0→1.0;
   - **传递闭包(多跳)物化**——把间接依赖路径 `A→B→C` 预先写进上下文,间接依赖 0.40→0.99;
   - **跨语言可抽取边**(Python↔C 扩展 `import _X`→`Modules/_X.c` 等)——实测 +1.0。
2. **检索侧投"深度/层级",不投"学习式检索器"**:rerank 调深(80)即大幅提升;
   学习式检索器(对比学习/RLCoder RL)实测**最多逼近静态分析、打不过**(RL 的 reward 本身 85% 指向静态算出的答案)。
3. **多跳依赖:文本化物化传递路径(B2)即可,不要上完整图数据库 RAG**——B3 完整图子图检索实测**劣于**轻量物化,且会丢节点属性。
4. **微调:仅用于"行为/格式对齐",不要用于"注入依赖知识"**——后者被 RAG 主导(大效应)且冗余,生成器/检索器/RL 全验证为边际或逼近上限。
5. **评测纪律(血的教训,强烈建议团队采纳)**:gen≠judge(被评模型≠裁判模型,否则 self-judge 静默虚高)、
   用例配对统计(不要拿重跑次数当样本量)、金标准与喂给模型的卡片解耦。我们在这上面栽过(被外部评委一刀点破 self-judge),修复后核心结论反而更硬。

## 2b. 脏依赖的关键工程建议(M30 实测,最可执行的一条)
**绝不让模型盲信依赖卡片。** 实测:静态难算的脏依赖(动态import/条件import/反射)上,
若 prompt 让模型"只信卡片",结构化表示**翻成负杠杆 −0.45**(错/缺卡片误导模型);
但改用**"承认静态分析可能不完整、结合模型判断、标注动态/条件依赖"的 prompt,完全救回(+0.84)甚至超 baseline**。
→ RepoMind 的依赖问答 prompt **必须**承认卡片可能不全、对动态/条件依赖留模型判断兜底,否则脏代码上有害。
**〔M33下修〕**:M31曾称humble"无条件免费升级",但M33中性措辞复测发现戏剧性效应部分是问题措辞artifact——中性提问下翻负基本消失、humble救回缩水且不跨裁判稳健。**收窄后的可执行建议仍成立但更朴素**:依赖卡片由静态分析生成、对动态/条件依赖会不全,prompt应承认这点、别让模型把卡片当全集——这条工程方向对,但量化收益是方向性的、非M31那么大。

## 2c. 脏依赖的产品建议(M38-40 原稿被 M41 大幅订正,以本订正版为准)
> ⛔ **M41 订正**:原 §2c"识破后必须触发再分析、别让模型自补(喂源码补全 90%)"经第六轮评委指控+从零去循环复查**失去支撑**——那个 90% 含挑口径 + 同源循环(58/58 片段含答案)+ 机理被推翻(模型不喂源码靠记忆也补 74%)。详见 `DIRTY_MECH_RETRACTION.md`。订正后能站住的只有下面两条:

- **能站住(E67,judge-independent 硬)**:humble prompt 让模型**识破**脏依赖(摆出"卡片可能漏、可能有条件/动态依赖"的审慎姿态)strict 7%→**99%**,跨 5 脏类型/4 语言稳健。**这是低成本的"警觉层",值得用**(补注:detect 是低门槛=采纳 humble 措辞,不等于模型真懂哪里脏)。
- **未能证明(原稿作废)**:"模型能否精确补出具体脏模块、再分析是否有用"——本实验**没有干净证据**:看似的补全能力大半是**对开源测试代码的记忆**(humble_prompt 不喂源码也 74%,但记的是 `_tkinter` 这类公开条件导入),**不会迁移到 RepoMind 私有代码**;喂源码的额外增益是同源循环 + 记忆,去循环后 ≈0。
- **诚实版决策语言**:① 依赖问答可用 humble 让模型保持审慎(识破真);② 但**不要据本项目就断言"模型能/不能补全私有脏依赖"或"必须再分析"**——这在私有代码上未实测,是开放问题;③ 真要让模型补出私有代码的具体脏依赖,大概率仍需把真实源码/动态分析结果喂进上下文(与项目主线一致),但**这是合理推断、非本项目已验证的结论**。

## 2d. 错误依赖卡片的下游代价(M43 实测:错卡片是危险负资产,不是小瑕疵)
你们的卡片不可能 100% 准。关键问题:**卡片错了之后,在下游任务里后果多严重?** 实测(下游=影响分析"改X→谁依赖需同步检查"):
- **错卡片(漏一个依赖方)→ 89% 危险结局**:模型把错卡片当权威**自信复述**不完整影响集,**静默漏报隐藏依赖方**(补出仅 7%、提示不全仅 2.4%、从不弃答)。工程师据此改动 → 漏改一个真依赖方 → 潜在 break。影响面越大越危险(大反向集 100% 危险)。
- **错卡片比"没有卡片"更危险**:无卡片时模型**弃答**("不确定")=0% 危险(工程师知道要查);错卡片把它变成自信漏报。**错卡片把"我不知道(安全)"变成"自信却不完整(危险)"。**
- **humble prompt 在下游兜不住**:识别任务里 humble 能识破脏依赖(99%),但下游错卡片**看起来是个正常完整列表(无可见脏线索)**,humble 审慎不触发——照抄不警告(humble 危险 94% ≈ strict 89%)。

**可执行建议**:
1. 对**安全攸关**的下游用途(改动影响面 / 删除安全性 / 接口同步同改),**绝不把静态依赖集当"权威完整"喂给模型或展示给用户**。
2. **别指望 humble prompt 兜底**——M45 控制实验证实更硬的结论:**humble 根本检测不出卡片是否不完整**(控制卡片长度后,它对"完整"和"缺项"卡片 hedge 一样多 Δ≈0,还对正确卡片误报警 32~54%)。它的 hedge 是随卡片长度/内容的反射、与真不完整无关,**不能当不完整守卫**;最危险的"长卡片漏几项"仍 44% 自信漏报。
2b. **别把模型的"谨慎措辞/免责声明"当安全信号**(M47 实测):模型在 ~99% 的回答里都附谨慎免责话,**危险决策和安全决策里的谨慎措辞率几乎相同(0.98 vs 0.99)**——免责话是无信息量样板,读它会被误导。**只看可执行裁决(它最终让你做什么),别看它说了多少"建议测试/可能不全"。**
3. **把"本集可能不完整(动态/条件导入依赖方未捕获)"做成结构性标注/UI**(不依赖模型自觉);危险下游动作**用完整性检查门控**,不靠模型自信。
4. 解法**不是**"不给卡片"(那 recall=0、全弃答),是"给卡片 + 明确标注不完整 + 不让模型断言完整"。
5. **〔M44→M45 精确化:漏报/误报取舍是分任务的〕** 漏报永远危险(下游自信漏真依赖方→改坏/误删)。误报的代价**随任务变**:在**影响分析**只是多查几个文件(浪费,可容忍→该任务偏高 recall、宁误报);但在**安全删除**误报会让模型对真安全删的文件 100% 误判"不安全"(过度拦截、该删不敢删)。→ **不要全局调一个 precision/recall:影响分析偏 recall,安全删除/重构两个方向都受伤、需高准确或对动作门控。**(E72)
**稳健性(M44 做厚)**:错卡片>无卡片的危险**跨 werkzeug/CPython/flask/gson(含 Java 跨语言)4 项目、跨 3 下游任务、跨错法程度**都成立——不是单设定偶然。
**真实幅度(M46→M48 最终定论,经口径区间 + 公平对比补全)**:
- 之前 0.85-0.9 是**人造随机漏边**;换真实 ast 实际漏掉的动态/反射依赖,危害**随判定口径在 ~1%~66% 区间**(中口径 ~18%;我之前单报 13% 偏有利,现公开全区间)。
- **conditional 不是真实静态盲点**(ast 抓得到 try/except),真实盲点是 **dynamic/reflection**。
- **"错卡片 > 无卡片"在公平对比下削弱**:无卡片的"安全"原本靠模型弃答(98%);真实工具强制作答时无卡片也危险(中 14%),**错卡片只是略更危险(中 +0.04/严 +0.28)**。
- **决策语言**:错依赖卡片仍是净负资产,但**真实增量 modest、量级随口径漂**——别当"几乎必然出事"。真正能省事的是:危险下游动作**用完整性门控**,既不靠模型自信、也不靠 humble、也不靠它的免责措辞(都不可靠)。详见 `DOWNSTREAM_HARM.md` §3f-3h。
**措辞(终审③)**:以上是"模型盲信**静态分析器输出**的下游后果",真值对着 ast 解析器非开发者真实依赖。
诚实边界:T2 安全删除问法下无卡片也有 21% 凭记忆断言"安全"。

## 3. 适用边界(诚实告知:方法何时失效)

**前提条件:依赖结构能被静态计算出来。** 能算出来 → 结构化表示数量级碾压(跨语言、跨规模在所测项目上一致成立)。
**算不出来 → 方法退化**,这些场景需要其他手段:
- C 的**宏展开**、跨编译单元的**隐式链接**依赖(不经 #include);
- 运行时**动态加载/绑定**(`importlib`、反射、插件机制、FFI 运行时);
- 构建系统**隐式链接**、生成代码、vendored 依赖。
- 超大规模(我们验证到 ~1456 卡片池/14×;**100×+ 上万卡片**可能需分层/层级检索,这是规模工程问题,未实测为硬边界)。

**与你们 RepoMind 私有场景的关系(诚实)**:本研究在开源项目(werkzeug/Go/Java/C/CPython/腾讯 rapidjson)上验证,
**不等于**你们私有代码库的真实场景,但提供了**最接近的旁证**——结论的机制(依赖是算出来的结构)是语言/项目无关的,
迁移到私有代码库时,关键看"你们的依赖结构能否被静态分析准确算出/覆盖"(见 §3 边界)。

## 4. 最硬的三个数据(撑起上面所有主张)
1. **结构化表示是数量级杠杆**:baseline 0.05 → 结构化卡片 0.94,**+0.89,p=1.5e-24**(CPython 工业级,独立裁判),跨 Go/Java/C 全 +0.6~0.93。
2. **模型侧穷尽边际**:PE/CoT 独立裁判不显著、微调 +0.04 冗余、RL 检索器 0.958<静态 1.000——**全部边际或逼近不超越**。
3. **可信度经得起复核**:两独立家族裁判 两LLM裁判间一致性 κ=0.806(LLM裁判间,非人工IRR)、去循环金标准、0 self-judge、外部评委三指控已逐条修复。

> 核心价值:这套研究帮你们**省下投错方向的钱**——别在更大模型/复杂 prompt/微调/图 RAG 上空耗,
> 把资源压到"依赖结构算准算全 + 检索找得到"上,这是经大效应证据支撑(主导杠杆,严打 4-5×/宽松 2×)、且原理上站得住的技术路线。

## 5. 〔M50-54〕co-change 负杠杆发现的甲方含义(带界,值得注意非强主张)

**用开发者真实共改(git 历史/bug 修复,独立于静态解析器)验证后的一个反直觉发现**:

> 在**静态分析看不见的变更耦合**边上(动态/运行时/插件装配等),给模型**正确的静态依赖卡片反而压低**它对这些边的召回——
> 卡片把模型锚定到静态邻居,挤掉了它本可凭语义召回的运行时共改伙伴。效应小而极稳(+0.05~0.11,跨 3 模型,p<1e-5)。

**对 RepoMind 的含义(谨慎)**:
- **动态耦合密集的场景**(插件系统、运行时注册、多后端动态分发——如你们若有类似 keras 三后端那种结构),
  **纯静态依赖卡片可能不只是"漏掉"动态共改,还会主动把模型从这些共改伙伴上带偏**。
- **可执行方向**:这类场景除静态卡片外,补**运行时/共改信号**(动态调用追踪、git 共改频率)进上下文,
  或对动态耦合区降低对静态卡片的依赖权重。
- **但务必带界,别当强主张**:① co-change≠依赖(是开发者行为代理,可能含流程噪声——虽实测被压低的边 97% 是核心源码耦合非噪声);
  ② 行为层确证非注意力层机制证明;③ 效应小;④ **evo 特异**——在"单次 bug 修复的真任务耦合"上此负杠杆**不出现(M54 实测 Δ=0)**,只在"高频历史共改"上显现。
- **定位**:**"值得注意、值得在动态耦合场景做产品级 A/B 的方向",不是"立即改产品"的强结论**。

**完整证据 + 无 GPU 复现**:`06_开发者真值发现/`(`verify_finding.py`)。co-change(非解析器开发者真值)方法论是真突破、被真外部评委从零复现。

> ⚠⚠ **M56 机制大幅下修(第 12 次修正)——上面这条甲方含义随之弱化**:干净对照(random_in_pool)证实"负杠杆"机制**主要是固定输出选择任务里的"机械挤占"(任何入池候选列表都挤占未列项的召回),非静态内容特异地带偏模型**(random 入池内容压低得一样多,p=0.94)。
> **故对甲方的可执行含义大幅收窄**:这更像"在'让模型从候选列表选'这种交互形态下,给一份不完整的候选清单会挤占模型对清单外项的召回"——是**交互/输出形态的工程注意点**,**不是"静态依赖卡片在原理上损害动态耦合理解"的强主张**。降级为"工程交互形态的注意点",**移出强建议**。
>
> ⚠⚠ **M57 再下修(第 13 次修正)——"模型能补静态盲区"也关掉**:用开放生成(绕开挤占)测发现,模型召回静态看不见的共改边**~94% 靠平凡信号**(读种子代码里已有的模块名 + 文件名相似 + 同包 __init__ 猜测),**真语义直觉补盲 ≤1.8% 且像项目记忆**。
> **故甲方不应指望"模型能凭语义补出静态分析漏掉的动态/运行时依赖"**——它主要在复述代码里已有的名字。补盲仍需把真实运行时/共改信号喂进上下文(与主线一致)。
> **co-change 方向对甲方的诚实净价值**:① 验证了"投静态分析准确性+检索深度"的主线(用正交真值);② 量化了静态盲区(~10-16% 真动态/运行时依赖任何静态分析都看不见,这部分**只能靠运行时/共改信号补,模型补不了**);③ 两个"漂亮"的产品含义(负杠杆、模型补盲)经诚实验证都不成立。

## 6. 〔M58〕静态盲区预警:用确定性 AST 检测器,不要用 LLM
你们若想"标注静态依赖可信度 / 在高风险处提示'依赖可能不全'",**用一个广义确定性动态-模式 AST 检测器**(覆盖 importlib/反射 getattr-setattr/entry_points/register 装饰器/__subclasses__/eval-exec),**不要用 LLM**:
- 实测让 Coder-14B 当盲区预警器:**过度预警(FP 率 0.29)、F1 0.755 < 一个 5 词关键词基线的 0.82**——LLM 不比确定性检测器准,还乱报。
- LLM 唯一的边(零样本覆盖未预设模式)对一个可维护的 AST 规则集不是优势;确定性检测器高精度 + 可扩展覆盖。
- **再次印证主线**:依赖/盲区是结构性事实,确定性计算 > 让模型猜——这次连"元层预警"也不例外。详见 `BLINDSPOT_WARNING.md`。

## 7. 〔M59〕提升 LLM 依赖能力的完整路径:静态卡片(主杠杆)+ 共改补充卡片(补盲区)
**可执行、实测有效**:静态分析有 10-16% 盲区(动态/运行时/插件),模型自己补不出(M57);但这部分**用 git 共改历史能算出来**,补进卡片后 **LLM 在静态不可见依赖上召回 18%→96%**(开放生成,非 token 伪影,visible 不退化)。
- **做法**:① 静态依赖卡片照旧(主杠杆);② 对静态盲区,用**高置信、去噪的 git 共改**(剔除 setup/version/doc/test 噪声)生成补充卡片,一并喂进上下文。
- **带界(诚实)**:这是"喂对结构→模型用起来"机制(模型读补充卡里的边),**非模型自己理解了隐藏耦合**;共改是行为代理(co-change≠依赖)、覆盖 10-16%、有噪声;端到端价值取决于共改卡算得准。安全攸关用途仍需结构性标注"可能不全"(`DOWNSTREAM_HARM`)。
- **一句话**:**别指望模型自己补盲区,但把共改算出的盲区边喂给它,它会用**——这是把主线杠杆从"纯静态结构"扩到"静态+共改结构"的可部署增量。详见 `COCHANGE_AUGMENT.md`。
- **⚠ 部署戒律(M61 实测,最重要的一条)——精度比覆盖更要命,宁可少补不可补错**:共改卡覆盖低只是少补盲区(召回≈0.16+0.82×覆盖率,损失有界、且有 0.16 内生底);但共改卡**含假边**时,**模型自信复述 64-83% 的假边**(confident wrong,等于给下游注入错依赖=负资产,见 `DOWNSTREAM_HARM`)。所以共改信号**必须高精度去噪**(高 support/confidence + 剔除 setup/version/doc/test 噪声);**宁可只补少数高置信盲区边,也不要喂噪声多的共改卡**。**别把 96% 当 KPI**:96% 是 card=gold 上限演示,真实部署召回 ≈ 你们共改信号对真依赖的覆盖率。

## 8. 一页纸:提升 LLM 代码依赖能力的完整可执行指南(M60 收口)
**核心:提升能力 = 喂确定性算出的依赖结构,不是改造模型。资源全投"结构准确性 + 检索深度 + 共改补全",别投模型改造/图 RAG/训检索器。**

**① 有效配方(照做,实测增益)**:静态依赖卡片(主杠杆,严打~4-5×/宽松~2×)+ 四旋钮(别名归一化 / rerank=80 / 多跳物化 0.40→0.99 / humble prompt)→ 能力拉到 0.94;**+ 共改补充卡片(高置信去噪)→ 静态盲区依赖召回提升(上限 96%=card=gold;〔M63 真实传递函数〕部署 recall≈0.16+0.82×共改有效覆盖率,随机/系统性退化几乎同,仍≫static 0.18;从未共改的硬边只得 0.16 底)。⚠ 别把 96% 当 KPI;共改卡必须高精度去噪——假边时模型自信复述 64-83%(下游负资产),精度比覆盖更要命**。

**PE 工程原则(M69 实测)**:幻觉类错误在**源头预防(System Prompt 反幻觉约束)> 末端纠正(后处理 filter)**——System Prompt 把编造文件名减 66%,filter 在预防后无可纠正(Δ=0)。优先把约束写进 System Prompt,而非堆后处理。

**② 无效路径黑名单(别投,各有实测证据)**:微调(LoRA/DoRA/IA³/DAPT,边际)、PE(System/few-shot/CoT,不显著)、图 RAG 子图检索(劣于轻量物化)、学习式检索器(≤静态)、让模型自补盲区(94% 平凡信号)、让模型当盲区预警器(不如确定性 AST 检测器)。**这份黑名单替你们省试错成本——每条都干净实验验过。**

**③ 边界与风险**:静态覆盖 ~84-90%;剩 10-16% 动态/运行时盲区靠共改/运行时信号补;**错误结构是负资产**(下游盲信→自信漏报,中口径~18%),安全攸关用途结构性标注"可能不全"、用完整性门控,不靠模型自觉/humble/免责措辞;co-change 是行为代理(≠依赖)、需去噪。

**④ 真值层诚实**:全部对静态解析器派生真值(parser-anchored);唯一非解析器验证来自 git 共改(开发者行为),验证了主线 + 量化了盲区。

## 9. 〔M68〕embedding 选型 vs rerank 的工程决策边界(实测掩盖度 11.3×)
**实测**:无 rerank 时 embedding 质量差异全传导(zh R@5 0.32 vs en 0.93,~3×);有 cross-encoder rerank 时差距被压到 0.05(**rerank 把 3× 差异压缩 11.3×**)。据此给你们的部署决策边界:

| 部署形态 | embedding 选型重要性 | 建议 |
|---|---|---|
| **带强 rerank**(cross-encoder 重排 top-N) | **次要旋钮**——rerank 兜底,弱/错语言 embedding 也救回(掩盖 11×) | 省心:embedding 用现成的即可;**但 rerank 有算力/延迟代价**(每 query 对 N 个候选做 cross-encoder 联合打分,N=30 量级) |
| **不带 rerank**(为省算力/降延迟,只用 embedding+ANN) | **大旋钮**——选错语言/模型检索直接崩(zh R@5 0.32) | **必须认真选型**:用英文/代码 embedding(bge-small-en/代码 emb),否则语义检索失效 |

**权衡建议**:① 在线低延迟/大流量场景倾向"不带 rerank"→ 把功夫花在 embedding 选型(选对语言/领域);② 离线/可容忍延迟场景倾向"带 rerank"→ embedding 选型可省心、把功夫花在 rerank 深度(主线 rerank=80 是最大杠杆)。
**rerank 兜底的边界(诚实,别当万能)**:rerank 只能把正确卡片从候选池 top-N 提到 top-K——**上限 = embedding 的候选池召回 R@N**;若 embedding 弱到正确卡片都进不了 top-N,rerank 也救不了。本任务即便最差 zh 也召进 top-30 的 93%,故救得回;更极端的错配可能击穿这个兜底。详见 `EVALUATION_REPORT.md` §3c。
