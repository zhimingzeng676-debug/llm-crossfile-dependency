# MEMORY_VS_STRUCTURE.md — M42:依赖知识里"参数记忆 vs 上下文结构"的占比(+私有代码迁移性)

> **入口(M41 撤稿废墟里的反主线张力)**:M41 发现模型不喂源码也能补 74% 开源条件导入(记住了公开代码)。
> 主线说"依赖在结构里不在模型里",数据却说"对见过的代码,依赖也在参数记忆里"。
> 本实验把这个张力做成可证伪的量化:模型答对依赖,多少靠**结构化卡片(上下文)**、多少靠**背过开源代码(记忆)**?
> 甲方价值:RepoMind 跑**私有代码**(模型没背过),若"碾压"含大块记忆,私有代码上会缩水——主线回答不了、甲方必须知道。
> **遵守 M41 防灌水红线**:全样本不挑子集、多口径全列、去同源、零 LLM 主指标、不把记忆当能力。

## 一、切分构造(任务一):标识符系统性改名
- **seen 组**:真实 werkzeug 文件名/模块名 + 问题点名"werkzeug 项目"(让参数记忆能 engage)。模型训练里见过 werkzeug。
- **unseen 组**:同一份依赖结构,**所有标识符确定性改名**(`exceptions`→`u15` 等,`build_memvsstruct.py` 无随机)+ 匿名"某 Python 项目"。**结构 100% 不变,只让"背诵"失效**。
- 唯一变量=模型是否背过这些名字。49 案例(forward 27/reverse 22),真值由 ast 独立依赖图导出(与 RAG 卡片解耦)。
- **披露局限**:用构造(b)改名;未用(a)训练截止后新项目(本地无)、(c)纯人造。改名只让**表层 token** 陌生,模型仍可能借依赖结构的统计模式;**不完全等于真实私有代码**(私有代码连结构习惯都可能不同)。

## 二、结果(任务二,judge-independent 关键词召回,全样本多口径)
| 口径(n=49) | baseline(abstain许可) | **baseline_forced**(禁abstain,强制猜) | full(结构化卡片) |
|---|---|---|---|
| **seen**(背过) | 0.000 | **0.169** | 0.986 |
| **unseen**(改名) | 0.000 | **0.017** | 1.000 |
| 差(seen−unseen) | 0 | **+0.152(p=2e-6)** | +(−0.014, ns) |

(forward/reverse 分口径一致:full 都 0.97~1.00、forced seen>unseen;全部写入 score_report.json,无挑子集。)

**读数**:
1. **切分有效**:强制猜测下 seen 17% vs unseen 2%——改名确实让记忆失效(**纯记忆贡献 +15.2pp,p=2e-6**)。
   (注:abstain 许可的自然 prompt 下 seen/unseen 都 0%——模型对"精确列出某文件的 import 全集"宁可诚实弃答也不硬猜,与全项目原始 baseline 近零一致。)
2. **结构化卡片 ≈ 满分且与记忆无关**:full 在 seen 98.6% / unseen **100%**——**改名(杀掉记忆)对卡片性能毫无影响**(迁移 −1.4pp,ns)。
3. **占比**:对"依赖列举"这个核心任务,**结构(卡片)≈100% ≫ 记忆(强制猜)≈15%**;且**结构对没背过的代码零损失迁移**。

> 透明:full≈100% 是因为依赖卡片**按设计就含 gold 依赖**(把结构喂进上下文=RAG 机制本身),非隐藏循环;baseline(无卡片)对照已隔离纯记忆,seen/unseen 隔离"是否背过"。本实验的信息量在 **baseline(记忆)弱且改名即灭、full(结构)强且迁移**,不在 full 的绝对值。

## 三、对主线的诚实再定位(任务三)
- **这是主线的稳健性/迁移性验证,不是独立新维度,也不修正主线**:主线"依赖价值在结构/上下文、不在模型"——本实验用改名做了干净对照,**坐实**了它:杀掉记忆(forced 17%→2%)不动卡片(100%→100%)。
- **解决 M41 张力**:模型**确有**开源依赖记忆(强制猜 ~15-17%;M41 的 74% 是"知名单个条件导入"这一窄现象),但这点记忆**又小又与卡片机制无关**——卡片有没有记忆都给 ~100%。**所以 M41 的"记忆"不威胁主线。**
- **不拔高(M41 教训)**:这**不是**能冲"优秀"的新发现,是一次**确认性**实验 + 一个对甲方有用的迁移数字。诚实说:它本质仍是主线复述,只是补了"私有代码迁移性"这个甲方关心的角。**结果不戏剧,如实报。**

## 四、对甲方(RepoMind)的结论(VALUE_FOR_REPOMIND 同步)
**好消息且可量化**:结构化依赖卡片的"碾压"**不靠模型背过你的代码**——在标识符全改名(模拟"模型没见过")的代码上,卡片性能 **100% 保持**(seen 98.6%→unseen 100%)。
- → **RAG/结构化卡片方案预期能干净迁移到 RepoMind 的私有代码**,不会因"模型没背过私有库"而缩水。
- **边界(诚实)**:本结论基于"改名"模拟未见代码(只灭表层记忆);真实私有代码的依赖**结构习惯**可能也不同,且本任务是"读卡片列依赖"(卡片含答案=RAG 机制)。**"卡片本身能否在私有代码上被静态分析算准算全"是另一回事**,仍取决于静态分析覆盖率(见适用边界),本实验不涉及。
