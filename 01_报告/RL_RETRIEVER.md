# RL_RETRIEVER.md — RLCoder式RL训检索器(M24-25:学习式检索器能打过静态分析吗)

> **动机**:对标 ICSE2025 **RLCoder**——不微调生成器,而用 RL 训练检索器,以**生成器对目标的 perplexity 下降**作 reward
> (无需标注,文献称跨语言泛化、超 SOTA)。M14 用对比学习微调检索器(−0.02 无效),但 RLCoder 是**不同范式**(RL+ppl)。
> **核心问题**:RL 学习式检索器能打过本项目的**精确静态分析依赖检索**吗?

## 0. 时间盒与截断规则(RL 易翻车)
- M24 末做收敛健康检查:reward 信号无效 / 训练不收敛 / 工程严重受阻 → **果断截断**,转"诚实记录尝试+预期结论"。
- RLCoder 不是必补的洞(M14 已有对比学习证据),是"锦上添花的封口"。绝不烧光剩余时间。

## 1. 设置(精简可行版)
- **数据隔离**:RL 训练查询从 flask/click/jinja2/requests 生成(34 条),**werkzeug 留作测试**。
- **每查询**:候选池 = 1 张静态分析正确卡片(该文件自身依赖卡片,含 import 列表)+ 5 张干扰卡片;目标 = 依赖答案文本。
- **reward**:Qwen-14B 对 target 的 **perplexity**(ppl 低 = 候选好)。这是 RLCoder 的核心无监督信号(`scripts/remote/ppl_reward.py`,vLLM echo+logprobs)。
- **红线**:gen≠judge(后续对照独立裁判)、单实例+锁、数据隔离、0 self-judge 自查。

## 2. 健康检查(M24 末:信号有效,不截断)
**关键诊断:argmin-ppl 候选 = 静态分析正确卡片?** → **85.3%(29/34)**;静态卡片 ppl 平均比干扰低 7.33(强区分度)。
**→ RLCoder 的 reward(ppl 下降)85% 指向静态分析算出的同一张依赖卡片**:RL 训练信号本身收敛向静态分析的答案,
即"RL 学习式检索器最多逼近静态分析"(那 14.7% 偏差是 reward 噪声=RLCoder 天花板)。**信号有效(非翻车),续做训练+对照。**

## 3. 四方对照(werkzeug 留出,纯 embedding 检索,gold=依赖卡片)
用 ppl 弱监督训 bge 检索器(34 正例,MNRL,`scripts/rl_train_eval.py`→models/bge-small-rl),werkzeug 24 查询:
| 检索器 | Recall@5 | MRR |
|---|---|---|
| 现成 bge-small | 0.583 | 0.309 |
| M14 对比学习(bge-small-zh-ft) | 1.000 | 0.941 |
| **RL(ppl 弱监督)** | **0.958** | **0.773** |
| **静态分析依赖检索(oracle)** | **1.000** | **1.000** |
- **RLCoder 有效**:ppl 弱监督把检索器从现成 0.583→0.958(确实学到更好检索器,非翻车)。
- **但 RL 打不过静态分析**:0.958/0.773 < 静态 1.000/1.000(甚至略低于 M14 对比学习)。

## 4. 结论:学习式检索器路径最终封口
- **RL/RLCoder ≤ 静态分析**(0.958<1.000)。ppl-reward 85% 指向静态答案 → **RL 收敛向静态分析、不超越**;静态依赖卡片是精确算出的 oracle,学习式检索最多逼近。
- **两条学习路径都封死**:M14 对比学习(端到端 −0.02)+ M24-25 RL(Recall 0.958<静态 1.000)——**"学习式检索器在依赖任务上 ≤ 静态分析"穷尽到最强 RL 路径**。
- **诚实定位**:这不是截断,是**做成的封口**(RL 收敛成功、且本身有效=远超现成 bge);它只是赢不了"精确计算"。**根因一脉相承:依赖是"算"出来的结构事实,计算它的静态分析是 oracle,学习去"猜"它打不过。**
- 边界:本实验为最小可行 RL(ppl 弱监督单轮蒸馏,非完整 REINFORCE 多轮);完整 RLCoder + 大数据可能再逼近静态分析,但上界是 oracle(1.000),无超越空间。
