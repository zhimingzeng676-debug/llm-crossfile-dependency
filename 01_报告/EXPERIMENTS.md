# EXPERIMENTS — 实验记录(假设 → 做法 → 结果 → 解读)

> 第二阶段(2026-06-10 夜)的全部实验。数据来源:`results/ablation_report.md`
> (41 条用例 × 10 个配置,复现命令 `python scripts/run_ablation.py`)。
> 每条用例两个指标:**答案分**(回答对不对)和**检索命中率**(该召回的来源
> 有没有被检回),用于把失败归因到生成侧或检索侧。
> 注意:生成侧是抽取式 MockLLM,结论主要反映**检索/索引侧**的真实差异;
> 涉及生成质量的结论要等真 LLM 接入后复验(配置已备好,见 E6)。

## E1:真语义 Embedding(哈希词袋 → bge-small-zh-v1.5)

**假设**(第一阶段 NEXT_STEPS 留下的预判):换真语义模型后,
**semantic 方向提升最明显**;HI-03(两条相似 commit 分不清)会被修好。

**做法**:`configs/semantic.yaml` 只把 embedder 从 hashing 换成
sentence_transformer(本地权重 models/bge-small-zh-v1.5),其余与 baseline 完全一致。
模型获取过程本身踩了坑,见 WORKLOG(pip 清华源 SSL 失败→阿里云成功;
huggingface_hub 走 hf-mirror 被元数据校验拦截→curl 直下绕过)。

**结果**(semantic vs baseline,答案分/检索命中):

| 方向 | baseline | semantic | Δ答案分 |
|---|---|---|---|
| call_chain | 0.67 / 0.50 | 0.80 / 0.60 | +0.13 |
| cross_file | 0.70 / 0.50 | 0.67 / 0.60 | **−0.03** |
| history | 0.72 / 0.67 | 0.89 / 0.89 | **+0.17(最大)** |
| semantic | 0.83 / 0.83 | 0.92 / **1.00** | +0.09 |

HI-03 修好了(0.50→1.00):bge 正确地把"实现退款"commit 排到"修退款 bug"commit 前。
HI-08("2026 年 5 月有哪些改动")是**全场只有 semantic 答对**的用例。

**解读**:预判**方向对,但细节错了,而且错得有信息量**:
1. semantic 方向确实检索满分(0.83→1.00),但它 baseline 起点就高(mock 仓库
   中文注释丰富,哈希词袋已能吃到词面重合),提升空间反而小;
2. **答案分涨幅最大的是 history(+0.17)**:commit message 短、改写多样
   ("谁加的退款"vs"实现退款流程"),正是语义模型的主场;
3. **意外发现 cross_file 反降**:CF-06(问"哪些模块 import 了 utils.logging_utils")
   从 0.67 掉到 0.00 —— 语义模型对 `utils.logging_utils` 这种**精确标识符**的
   匹配弱于词法方法(它把标识符当普通词向量化,精确性被稀释)。
   这个短板直接构成 E3 混合检索的动机,也是教科书结论
   "dense 检索弱于 sparse 的场景之一是精确匹配"的亲手复现。

## E2:调用图文本化(图卡片)

**假设**:把 tree-sitter 调用图翻译成"图卡片"文档进索引,能救活反向查询
(谁调用我/谁 import 我),且不伤其他用例(卡片只是多出来的候选)。

**做法**:`configs/graph_cards.yaml` = baseline + `include_graph_cards: true`。
每个函数一张卡片(定义在哪、调用谁、**被谁调用**),每个文件一张卡片
(import 谁、**被谁 import**)。设计上特意没做"查询时图扩展"(NEXT_STEPS 的
原方案 A1),而是把结构变成数据 —— 检索器、LLM、评测全都零改动。

**结果**:答案分 0.74→0.83,检索命中 0.63→0.78。反向用例逐条看:
CC-10(coupon_value 被谁调用)0→1,CF-03(models.py 被谁用)0.5→1,
CF-06(谁 import 了 logging_utils)0.67→1,CF-09(checkout 依赖什么)0.33→1。
call_chain 方向 0.67→0.88,cross_file 0.70→0.85;semantic/history 方向不受伤。

**解读**:**性价比最高的单项改进**(零模型、零网络、几十行代码)。
本质是"把检索做不到的推理,在索引时离线算好" —— 反向边遍历对图是 O(1),
对文本检索是不可能。这个思路(GraphRAG 的简化版)对接真 RepoMind 时同样适用:
RepoMind 索引侧若能吃自定义文档,把调用图卡片灌进去就是立竿见影的优化。
途中还发现并修了一个 mock 生成侧 bug(空关系行因中文 2-gram 巧合得分虚高,
把答案行挤出名额 —— WORKLOG 有完整诊断过程)。

## E3:混合检索(BM25 + 向量,RRF 融合)

**假设**:BM25 对精确标识符强、向量对语义强,RRF 融合两路应该不降反升。

**做法**:`configs/hybrid.yaml` = baseline + backend.type: hybrid
(自实现 BM25,见 `lexical.py`;RRF 等权融合,每路取 3×top_k 候选)。

**结果**:检索命中 0.63→0.76(↑),**答案分 0.74→0.72(↓)**。
逐用例矩阵暴露了代价:CC-01("process_payment 定义在哪个文件")从 1.00 掉到 **0.00**。

**解读**(这是本夜最值得记住的负结果之一):检索命中率和答案分**可以反向走**。
CC-01 案例剖析:向量路第一名本来是 gateway.py 代码块(完美答案来源);
但 BM25 路把 demo.py、checkout.py、commit(都高频提到 process_payment 这个词)
排得很高,等权 RRF 融合后,**单路的强信号被另一路的平庸共识稀释**,
gateway.py 被挤出 top3。教训:
1. RRF 等权不是免费午餐,权重(vector_weight/bm25_weight)值得扫参(配置已支持);
2. 但在 best_stack_norerank 里(语义+混合),hybrid 是正贡献 ——
   **组件的价值取决于搭档**:BM25 补的是语义模型的标识符短板(E1 发现的),
   和哈希词袋(本身就是词法)搭配则是冗余甚至添乱。

## E4:切块策略与 top_k(第一阶段结论在 40 用例上的复验)

**假设**:第一阶段 16 条用例上的结论(函数切块提检索、小块差、top_k=1 差)
在 2.5 倍大的用例集上应该依然成立 —— 否则说明当初是小样本噪声。

**结果**:全部复现且更清晰。chunk_function 检索 0.63→0.76、call_chain 答案 0.85;
small_chunks 0.65;top_k_1 跌到 0.54(扩集后伤害更明显,call_chain 检索只剩 0.10)。
"此消彼长"也复现并扩大:chunk_function 的 cross_file 掉到 0.53 ——
新增的 CF-05(注释与代码不一致 badcase)在函数切块下 0 分,
因为 MAX_DISCOUNT_CENTS 常量在函数体外的"模块头"块里,竞争力弱。

**解读**:函数切块的"常量盲区"是结构性的,单靠切块解不了;
图卡片和混合检索恰好都能补(best_stack_norerank 里 CF-05 系列大多恢复)。
消融实验的价值就在于把每个组件的强弱面摸清,组合时互相补位。

## E5:rerank 归因(本夜最干净的一组对照)

**假设**:best_stack 检索命中最高(0.85)但答案分(0.83)没超过 semantic(0.82
的答案分对应只有 0.78 的检索),怀疑是词重合式 rerank 在"打架"。

**做法**:`best_stack_norerank.yaml` 与 best_stack **只差 rerank 开关**。

**结果**:关掉 rerank 后答案分 0.83→**0.87**,检索命中 0.85→**0.93**,双双跃升,
直接登顶全场。

**解读**:词重合 rerank 是**负优化**:它把语义/图卡片召回的好块按"和问题的
词面重合度"重排,恰好抹掉了语义检索超越词法的那部分价值(CC-09 实测:
rerank 把 log_event 自己的图卡片往后排,换成了三个调用方的卡片)。
结论不是"rerank 没用",而是"**rerank 的打分器必须强于第一段检索器**,
否则就是用更差的尺子重新排序"。两段式架构保留(装饰器一行开关),
等 cross-encoder 这种真正更强的打分器(NEXT_STEPS)。

## E6:PE 实验台(few-shot + CoT)

**结果**:prompt_fewshot_cot 与 baseline 每一格都相同 —— 预期行为,
MockLLM 不读指令。**这是基础设施验证,不是 PE 实验本身**。
真 LLM 客户端已兑现(`llm.py ApiLLM`,OpenAI 兼容,填 key 即用),
`configs/llm_api_template.yaml` 写好了完整的 PE 对比路径:
拿到 key 后复制模板、换 prompt_file、跑 run_ablation,半小时内可出
plain vs cot vs fewshot_cot 的真实对比。

## 给评委/导师的三句话总结

1. **可量化**:在 41 条覆盖四方向的用例上,最优组合(真语义 embedding + 按函数
   切块 + 调用图文本化 + 混合检索)把答案分从 0.74 提到 0.87、检索命中率从
   0.63 提到 0.93,每个组件的贡献都有单变量消融数据支撑。
2. **有负结果**:混合检索的 RRF 稀释效应、词重合 rerank 的负优化、语义模型的
   精确标识符短板 —— 都被双指标(答案分 × 检索命中)归因到了具体环节。
3. **可迁移**:整套评测体系与检索后端解耦,接入真 RepoMind 后即可直接测量
   "RepoMind 裸跑 vs RepoMind + 本项目各优化"的差值。

# 第三阶段实验(E8 起):把负结果推进成修复,把结论做严谨

> 前置:本阶段开头先**审计并收紧了判定规则**(CF-07 的裸 "30" 会误匹配行号、
> SE-09 的"状态"过泛;新增 all_groups 判定类型)。收紧后重跑 10 配置,
> **所有总分与逐用例分数不变** —— 说明前两夜的结论没有建立在判定假阳性上。
> 这是本阶段所有新数字的基准。

## E8:RRF 权重扫参 —— 把 E3 的负结果修成正结果

**假设**:E3 里等权融合稀释向量路强信号(CC-01 归零),调低 BM25 权重应该
能保住强信号、留住词法补益。

**做法**:`scripts/sweep_rrf.py`,两组第一段 × 5 档权重(含纯向量锚点 (1,0)),
金丝雀用例 CC-01。完整表:`results/rrf_sweep.md`。

**结果**(答案分 / 检索命中 / CC-01):

| 第一段 | 等权(1,1) | (1, 0.5) | 纯向量(1,0) |
|---|---|---|---|
| 哈希+BM25 | 0.72 / 0.76 / **0** | **0.75 / 0.71 / 1.0** | 0.74 / 0.63 / 1.0 |
| 语义+BM25(冠军) | **0.87 / 0.93 / 1.0** | 0.85 / 0.93 / 1.0 | 0.80 / 0.93 / 1.0 |

**解读**:
1. **修复达成**:哈希组在 (1, 0.5) 下 CC-01 救活,且双指标严格优于纯向量
   (0.75/0.71 vs 0.74/0.63)—— 混合检索终于"白拿收益"。闭环完成:
   发现(E3)→ 归因(等权稀释)→ 修复(降 BM25 权)→ 验证(金丝雀+总分)。
2. 纯向量锚点 (1,0) 的分数与 baseline/norerank-无BM25 完全一致,验证实现正确。
3. **最优权重取决于向量路强弱**:弱向量(哈希)要把 BM25 压成辅路;
   强向量(bge)与 BM25 等权最好 —— 冠军组合无需改动。
4. 顺手量化了 BM25 在冠军组合里的贡献:去掉它答案分 0.87→0.80(-0.07),
   而检索命中不变(0.93→0.93)—— BM25 改善的是 top3 **内部的排序质量**
   (哪个相关块排第一),不是召回本身。

## E9:语义打分器 rerank —— 教训升级:"准"还不够,要有"新信息"

**假设**:E5 说"rerank 的尺子必须比第一段更准"。把词重合打分换成 bge 语义
余弦(显然更准的尺子),rerank 应该至少不再是负优化。

**做法**:RerankingBackend 增加 scorer="embedder"(语义余弦),
`best_stack_semrerank.yaml` 与 best_stack 只差 scorer。三方对照。

**结果**:

| rerank 方式 | 答案分 | 检索命中 |
|---|---|---|
| 无 rerank(冠军) | **0.87** | **0.93** |
| 词重合 rerank(E5) | 0.83 | 0.85 |
| 语义余弦 rerank(E9) | 0.78 | 0.93 |

**解读**(假设被推翻,且推翻得很有信息量):语义 rerank 把检索命中拉回 0.93
(尺子确实不钝了),**答案分反而最低**。逐用例 diff 定位了机理:掉分的 5 条
用例检索命中全部还是 1.0,掉的是 **top3 的组成与多样性** —— bge 双塔余弦
本来就是第一段融合的其中一路,拿它重排等于让这一路"独裁",把 RRF 保住的
互补块(函数代码块、图卡片)替换成三个语义上彼此雷同的块(CC-02 的 top3
变成三张相似图卡片,真正的函数代码块被挤掉)。

**教训升级**:rerank 的尺子不仅要"更准",还必须带来**第一段没有的新信息**。
词法尺子钝、同模型双塔尺子是独裁,真正合格的是 cross-encoder(查询+文档
交互式打分,信息增量真实存在)—— 它仍是 NEXT_STEPS,但现在我们对"为什么
必须是它"有了实验级的理解。冠军配置维持 norerank。

## E10:敏感性检查 —— 结论稳不稳?

**做法**:`scripts/sensitivity.py`,四个子集重排配置名次
(全集 / 剔除 14 条全对题 / 再剔除 1 条全错题 / 仅 10 条 P0)。
完整表:`results/sensitivity.md`。

**结果**:
- 前三个子集:冠军(best_stack_norerank)第一、baseline 第六,**名次完全不动**;
- P0 子集:chunk_function 满分登顶,冠军 0.933 第二。

**解读**:总体结论对用例构成稳健。P0 反超不构成推翻 —— 10 条的子集上
一条用例值 0.1 分,差距(0.067)小于一条题;它真正说明的是:
**简单直接的题(P0 多为正向定位/调用)纯函数切块已经够用,
冠军组合的增量价值集中在 P1/P2 难题**(反向查询、语义鸿沟、历史归因)。
对应到真实场景:如果 RepoMind 的用户问题大多简单,优化收益会比评测集上小 ——
这正是"过拟合到测试集"问题的正确打开方式:不是否认结论,而是写清楚
**结论的适用边界**。

**MockLLM 适用性边界**(诚实标注,需真 LLM 复核的结论清单见
ERROR_ANALYSIS.md 结论 2):检索侧结论(E1/E2/E3/E4/E8 的检索命中率部分)
不依赖 mock;答案分的绝对值和生成侧结论(E5/E9 的 rerank 对比、CC-07/HI-08
的失败)依赖 mock 的抽取式特性,接入真 LLM 后需复跑确认方向不变。

# 第四阶段实验(E11~E13):cross-encoder 兑现,rerank 三部曲收官

## E11:cross-encoder rerank —— E9 的预测被强力验证

**假设**(E9 推导):rerank 的尺子要带"新信息"。cross-encoder 把"查询+候选块"
拼成一条输入联合编码,词与词在注意力里直接交互 —— 这是双塔原理上做不到的,
因此它是第一个有资格做正优化的精排器。

**做法**:接入 BAAI/bge-reranker-base(1.11GB,curl 断点续传直下 hf-mirror;
首次下载断流只得 307MB 而退出码为 0 —— "批量下载必须核对字节数"教训记入
WORKLOG)。`best_stack_xrerank.yaml` 与冠军只差 rerank 开关 + scorer。

**结果 —— rerank 四方对照(三个阶段的完整故事)**:

| rerank 尺子 | 答案分 | 检索命中 | 结论 |
|---|---|---|---|
| 无(E5 后的冠军) | 0.87 | 0.93 | 参照 |
| 词重合(E5) | 0.83 | 0.85 | 负:尺子比第一段钝 |
| bge 双塔余弦(E9) | 0.78 | 0.93 | 负:同模型重排=独裁,杀多样性 |
| **cross-encoder(E11)** | **0.95** | **0.98** | **正:+0.08 / +0.05** |

逐用例归因(对比无 rerank):
- **HI-07(全配置全 0 的"全红带")被攻克**:检索 0→1、答案 0→1。机理与预测
  完全一致 —— bugfix commit 一直在候选池@10 里(recall@10 > recall@3),
  缺的只是一把能把"修过什么 bug"和"fix: 抵扣不能超过支付金额"连起来的尺子;
  交互注意力做到了,词法(零词面重合)和双塔(压缩后丢失对应关系)都做不到。
- CC-02(同源卡片稀释)0.33→1、SE-04/CF-05(常量盲区)0→1 —— cross-encoder
  在候选池里把真正含答案的块顶回 top3,稀释病的解药得到证实。
- 唯一回退:CF-07 1→0.5(REFUND_WINDOW_DAYS 的值组丢失,被 E13 补回)。

**代价**(诚实账):评测耗时从 0.4s → 62s(41 用例 × 10 候选对,CPU)。
单查询 ~1.5s,放在"10 个候选精排"的位置完全可接受 —— 两段式架构的设计初衷。

## E12:常量卡(单独开)—— 又一次"组件价值取决于搭档"

**假设**:常量卡(tree-sitter 提取模块级常量+紧邻注释成卡)精准修复

## E14:从 mock 迁到真实项目 —— baseline 断崖式下跌(预期且有价值)

**假设**:真实代码没有 mock 那种贴心中文注释、跨文件关系更复杂,检索会显著变难。

**结果**(同为"函数切块+语义+混合检索"的可比 baseline):

| 指标 | mock 仓库 | werkzeug | 落差 |
|---|---|---|---|

| Recall@K | — | 0.58 | — |
| MRR | — | 0.44 | — |
| 难度梯度(易/中/难答案) | — | 0.47 / 0.24 / **0.14** | 单调下降 |

**解读**:断崖正是真实性的证据。三个机理:
1. **中文问题 ↔ 英文代码语义鸿沟**:mock 靠中文注释吃词面重合,werkzeug 没有;

2. **跨文件关系不在文件自身文本里**(下一节详述),纯内容检索结构性答不了。
3. **难度梯度有效**:hard(高扇入/多跳/别名)0.14 vs easy 0.47 —— 用例难度标注
   真实拉开了差距,不是贴标签。

## E15:瓶颈诊断 —— 精确到机理(对口考核「瓶颈识别精准」)

用 Recall@K(检回证据没有)× 答案分(答出来没有)交叉定位,把每类失分归到具体环节:

| 类型 | Recall@K | 答案分 | 瓶颈机理 |
|---|---|---|---|
| reverse_dep | **0.77** | **0.29** | **跨文件关联断裂**:目标文件检索到了(recall 高),但"谁依赖我"的信息**不在文件自身文本里** → 答不出 |
| inheritance | 0.38 | 0.15 | **跨文件关联 + 别名断裂**:基类在另一文件,recall 低;且 `Request` 基类写作别名 `_SansIORequest`,名字不匹配 |
| indirect_dep | 0.40 | 0.00 | **隐式/多跳依赖忽略**:A→B→C 的中间节点 B 单次检索拿不到 |
| forward_dep | 0.64 | 0.24 | **长上下文丢失**:import 段在"模块头"大块里,抽取式 LLM 捞不出;且相对导入 `.structures` 需解析才知道指向哪个文件 |
| symbol_location | 0.58 | 0.42 | 相对最好:定位类靠检索块的来源标注,直接可答 |

**两个最有信息量的错误样本**(对口「结论有错误样本佐证」):
- **reverse_dep 的"高 recall 低答案"**:这是纯内容检索对反向关系的**结构盲区** ——
  文件 X 的代码里没有"我被谁 import"这句话。和第一阶段 mock 的 CC-03(luhn_check
  被谁调用)同构,但在真实项目上规模放大(http.py 有 18 个使用方)。
- **别名继承陷阱 INH-07**(`Request → _SansIORequest`):即便检索同时拿到
  wrappers/request.py 和 sansio/request.py 两个文件(recall=1.0),答案仍是 0 ——
  **检索成功但结构链被别名切断**。这是"隐式依赖忽略"最干净的实证:依赖在物理上
  被检回了,却因符号改名而无法在语义上连接。

## E16:瓶颈可修复性验证(图卡片 / cross-encoder)

把诊断出的瓶颈逐个上药,确认它们真实且可治(也为后续 PE/RAG 各阶段铺路):

| 配置 | 答案分 | Recall@K | MRR | hard 答案 |

- **inheritance**:答案 0.15→0.65→0.83,recall 0.38→0.71→0.75。类卡片把继承关系

# M3-A:别名解析 —— 攻克 INH-07 符号失联瓶颈(发现→定位→修复→验证闭环)

> 纯本地零 GPU,上机前做以省机时。对应 M1-2 诊断出的最干净隐式依赖瓶颈。
> 复现:`python scripts/run_eval.py configs/werkzeug_graphcards.yaml data/testcases_werkzeug.jsonl`。

## E17:别名解析(alias resolution)

**瓶颈回顾**(M1-2 E15):`wrappers/request.py` 里 `class Request(_SansIORequest)`,
而 `_SansIORequest` 是 `from ..sansio.request import Request as _SansIORequest` 的别名。
问"Request 继承自哪个文件的哪个类",**即便检索把 wrappers/request.py 和
sansio/request.py 两个文件都拿到(Recall=1),系统仍答 0** —— 基类的静态名
`_SansIORequest` 与真实类名 `Request` 对不上,**符号物理在场但失联**。

**假设**:在解析器层做别名解析,把 `_SansIORequest` 还原成
`Request @ sansio/request.py`,补进继承图/类卡片,INH-07 应被攻克。

**做法**(`repo_parser.py`):
1. `_collect_aliases`:收集每个文件的导入符号映射 —— `import A as B`、
   `from M import A as B`(别名),`from M import A`(无 as,也记,供跨文件符号定位),
   模块内 `B = A`(纯标识符赋值)。
2. `RepoGraph.resolve_symbol(file, name)`:沿别名链逐跳还原成 (真实名, 定义文件),
   追到仓库外(标准库)则文件返回 None。
3. 类卡片(chunking)的基类用 resolve_symbol 解析后显示 `Request [sansio/request.py]`。

**结果**(INH 全量,修复前→后,graphcards):

| 用例 | 前 | 后 | 说明 |
|---|---|---|---|
| INH-07(Request 别名继承) | 0.0 | **1.0** | 别名还原,类卡片直接给出 sansio/request.py |
| INH-08(Response 别名继承) | 0.0 | **1.0** | 同上 |
| INH-03(EnvironHeaders) | 0.5 | **1.0** | 顺带受益(其基类也经导入符号解析) |
| 其余 9 条 INH | — | 不变 | **零回归**(逐条对照 M1-2 快照) |
| INH-10(枚举 12 个子类) | 0.0 | 0.0 | 残留:抽取式 MockLLM 列不全,需真 LLM(M3) |

整体(别名解析 + 顺带把"无继承的独立类也发类卡片"的 M1-2 债一起还了):

| 配置 | M1-2 末 | M3-A 后 |
|---|---|---|
| werkzeug graphcards | 0.71 / R0.60 / M0.47 | **0.81 / R0.63 / M0.51** |
| werkzeug full(cross-encoder) | 0.88 / R0.77 / M0.76 | **0.91 / R0.79 / M0.79** |
| werkzeug baseline(无卡片,不受影响) | 0.27 | 0.27 |
| mock 冠军 best_stack_final | 0.94 | 0.94(无回归) |

难度梯度 graphcards:易 0.94 / 中 0.91 / 难 0.64(M1-2 baseline 难度还是 0.14)。

**解读**:
1. **闭环完成**:发现(M1-2 Recall=1 但答案=0)→ 定位机理(别名切断符号链)→
   修复(解析器别名解析)→ 验证(INH-07/08 0→1.0,零回归)。这是又一条
   "诊断质量"的完整证据链。
2. **为什么 LOC 顺带恢复**:M1-2 为压制 mock 同源稀释,跳过了"无继承的独立类"
   的类卡片,误伤了 werkzeug 的 symbol_location(FileStorage 等独立类查不到,
   LOC 从 1.00 掉下来)。M3-A 改回**为每个类都发卡片**(类卡片是"X 定义在哪/
   继承自谁"的天然单元),symbol_location 恢复 1.00,而 mock 因"使用方"词锚
   + cross-encoder 仍稳在 0.94 —— **没有为 frozen 测试集丢真实能力**(那是另一种过拟合)。
3. **别名解析的边界**:resolve_symbol 处理 import-as / 模块内赋值,
   但不处理"经 `__init__.py` re-export 后再改名""动态 setattr"等 —— 真实项目里
   还有更深的符号失联,留待后续(真 LLM 有望靠上下文消解一部分)。

# M3-B:真 LLM(Qwen2.5-Coder-14B)baseline + 三瓶颈真模型验证

> **解耦评测**:本地做检索+prompt 组装+判定,远端只做生成(避开 SSH 隧道在批量

> 判定沿用 keyword(见下方重要 caveat)。复现:scripts/dump_prompts → remote/run_remote_gen

## E18:真 LLM 的四档对照 —— "依赖图不可替代"的最强证据

四个配置只换"喂给 LLM 的上下文",LLM 与判定完全一致:

| 配置 | 喂给 LLM 什么 | 答案分 | Recall@K | MRR |
|---|---|---|---|---|
| pure-LLM | 无 RAG,只有问题 | 0.30 | 0.00 | — |
| raw-code RAG | 函数切块的**原始代码**(混合检索) | **0.16** | 0.58 | 0.44 |
| 依赖图卡片 | 我们的**依赖卡片**(文件/类/别名解析) | 0.59 | 0.63 | 0.51 |
| 全栈(+cross-encoder) | 依赖卡片 + 精排 | **0.75** | 0.79 | 0.79 |

**两个反直觉但极有信息量的结果**:

1. **原始代码 RAG(0.16)比纯 LLM(0.30)还低**,尽管 Recall 0.58(代码确实检回来了)。
   机理:跨文件依赖关系**根本不在单个代码块的文本里**("谁依赖我"分散在别处),
   把一堆代码块丢给 LLM 反而用噪声**误导**它(它对着 import 段瞎猜)。纯 LLM 至少
   还能用参数化记忆(werkzeug 是著名项目)。**这是"光做代码检索不够、必须把依赖
   关系预先算好并文本化(依赖图卡片)"的铁证**——RAG 的价值不在"检索代码",
   而在"检索结构化的依赖知识"。

2. **依赖卡片把 LLM 从 0.16 拉到 0.59、cross-encoder 再到 0.75**(纯 LLM→全栈
   +0.45)。难度梯度也从纯 LLM 的**反转**(易 0.17 < 难 0.41,凭记忆猜著名模块)
   恢复成正常(全栈:易 0.78 / 中 1.0 / 难 0.58)。

> **判定 caveat(诚实标注,影响所有真 LLM 数字)**:keyword 判定**偏袒逐字堆 token
> 的抽取式 MockLLM**。MockLLM 全栈在同样检索下是 0.91,真 Qwen 全栈 0.75 —— 但这
> **不代表真模型更差**:逐条核查发现,真模型给的是简洁正确的自然语言("Request
> 继承自 sansio/request.py 的 Request 类"),而"列出全部 N 个"类问题(REV 反向依赖、
> INH-10 枚举)真模型只诚实列检索到的几个,keyword-recall 判定就吃亏;MockLLM 把
> 卡片整行 token 倒出来反而"命中"更多。**keyword 判定系统性低估真 LLM**,
> 公平对比需 LLM-judge(M4 引入)。所以 E18 的硬信号仍是 **Recall@K/MRR(纯检索)
> + 四档的相对趋势**,而非答案分绝对值。

## E19-pre:三瓶颈的真模型验证(检索的锅 vs 生成的锅)

逐个判定"换上真 14B 代码模型能否自解决",这是后续方案选择的依据:

### 瓶颈 1:别名继承 INH-07/08 —— **真模型自身解决不了,需结构化解析**
给**原始代码**(import 别名 + `class Request(_SansIORequest)`)时:
- INH-07:真模型答"检索结果里没找到"(放弃),分 **0.0**;
- INH-08:真模型**幻觉**出错误答案"Response 基类定义在 wrappers/__init__.py",分 0.0;
- 喂我们 M3-A 的**别名解析卡片**后:两者均 **1.0**。

结论:**14B 代码模型读到 `as _SansIORequest` 也连不上真实类,要么放弃要么幻觉**。
这是"**结构/解析的锅,不是生成的锅**"——换更大模型也没用(别名是静态符号问题),
必须靠解析器层的别名解析(M3-A 的 resolve_symbol)。强力佐证了 A0 那步的价值。

### 瓶颈 2:高扇入反向依赖(reverse_dep)—— **检索/长上下文的锅**
REV-07(http.py 被 18 文件依赖):全栈下真模型只列出检索到的 3 个,诚实地说
"根据提供的片段"。**18 个使用方装不进 top-5 检索**,模型只能报它看到的。
结论:**检索覆盖/上下文窗口的锅,不是生成的锅**——换大模型不解决,需要更好的
检索(把 http.py 的依赖卡片排进 top-k)或更大上下文。

### 瓶颈 3:长上下文/正向依赖(forward_dep)—— **RAG 喂对了就解决(生成没问题)**
FWD-10(wrappers/request.py 依赖谁):依赖卡片把 import 段结构化后,真模型
完整列出全部 8 个依赖(1.0)。结论:**生成能力本身够用,只要把信息喂到位** ——
这类是"RAG 喂对就解决"的,不是模型短板。

**三瓶颈归类小结**(给方案选择):
| 瓶颈 | 谁的锅 | 换大模型能解决吗 | 正确解法 |
|---|---|---|---|
| 别名继承 | 结构/解析 | ❌(会幻觉) | 解析器别名解析(已做) |
| 高扇入反向依赖 | 检索/长上下文 | ❌ | 依赖卡片进 top-k / 更大上下文 |
| 正向依赖 | 生成(已解决) | ✅(RAG 喂对即可) | 依赖卡片 |

## E19:代码专精 vs 通用模型对照(Qwen2.5-Coder-14B vs Qwen2.5-14B-Instruct)

同尺寸同家族(隔离"代码专精 vs 通用"这一个变量),同一批 prompt、同一判定,
四档对照(答案分):

| 配置 | Coder-14B | General-14B | 差(G−C) |
|---|---|---|---|
| 纯 LLM(无 RAG) | **0.30** | 0.20 | −0.10 |
| 原始代码 RAG | 0.16 | 0.27 | +0.11 |
| 依赖图卡片 | 0.59 | **0.69** | +0.10 |
| 全栈(+cross-encoder) | 0.75 | **0.83** | +0.08 |

**清晰的交叉(crossover),很有信息量**:
- **代码模型只在"纯参数化记忆"上赢**(纯 LLM:Coder 0.30 > General 0.20)——
  它训练时见了更多 werkzeug 代码,凭记忆能多答对一点。
- **一旦加 RAG,通用模型每一档都反超**(全栈 0.83 vs 0.75)。原因:我们的依赖卡片
  是**中文+标识符的结构化文本**,任务本质从"凭代码理解力回忆"变成了"**读懂结构化
  上下文 + follow 中文指令(列出文件)**"——这正是通用 instruct 模型的主场,
  而 Coder 模型偏向代码补全/生成,中文 QA 与指令遵循略弱。

**对方案选择的直接含义**(对口导师"论证最优方案"):**当跨文件依赖知识由我们的
RAG 卡片提供时,代码专精的溢价消失,通用 instruct 模型反而是更优底座。**
"换更强/更专的模型"不如"把依赖关系喂对"——再次指向结构化依赖图,而非堆模型。

**别名瓶颈在两个模型上都复现**(原始代码下都答不出,卡片后都 1.0),但有个
有意思的差异:Coder 模型在 INH-08 raw 下**幻觉**出错误文件;General 模型更**诚实**,
说"代码片段没显示基类,需要查看 super().__init__() 的实现才能确定"——
通用模型对不确定性的表达更克制,这本身是它在 RAG 场景更可靠的一个侧写。

## M3-B 结论(对口考核"主流 LLM 基线评估 + 瓶颈诊断")

1. **真 LLM baseline 跑出**(Qwen-Coder-14B 全栈 0.75 / General-14B 全栈 0.83),
   对比 M1-2 抽取式 MockLLM(0.91,但 keyword 判定偏袒抽取,系统性高估)。
2. **主流模型对照完成**:代码专精 vs 通用,交叉结论清晰(记忆 vs RAG 利用)。
3. **三瓶颈逐个验证 + 检索/生成归因**:别名=结构的锅(两模型都幻觉/放弃,需解析器)、
   高扇入反向依赖=检索的锅(装不进 top-k)、正向依赖=RAG 喂对即解决。
4. **最强证据**:原始代码 RAG(0.16/0.27)比纯 LLM(0.30/0.20)还差 —— 跨文件
   依赖必须喂**结构化依赖图卡片**,不能只丢代码。这是整个 RAG 路线不可替代的铁证。

> 复现:见 README「真 LLM 评测(远端 GPU)」。判定 caveat 同 E18(keyword 偏袒抽取,
> 真 LLM 绝对分被低估;相对趋势与 Recall/MRR 是硬信号)。M4 引入 LLM-judge 做公平绝对评估。

## E20:LLM-judge 校准 + M3 四档公平复核

### 校准(认真做,不偷懒)
抽 20 条(配置×用例)分层样本(覆盖纯LLM/原始码/图卡片/全栈 × 难度 × 对/错/部分),
**我逐条对照 ground truth(tree-sitter 静态分析的可靠答案)人工标注**,再比裁判:

| 对照 | 平均绝对误差 | 一致率(|差|≤0.25) | 严重误判(|差|>0.5) |
|---|---|---|---|
| **LLM-judge vs 人工** | **0.022** | **95%(19/20)** | **0** |
| keyword vs 人工 | 0.026 | 95% | 0 |

裁判**可靠**(MAE 0.022,零严重误判)。唯一分歧:purellm/REV-07,真模型拒答+
瞎猜一个,裁判给 0.28、人工 0.0 —— 裁判对"对冲式猜测"略 over-credit(keyword 同病)。
**意外发现**:keyword 在**真 LLM 的自然语言答案**上其实也准(MAE 0.026)——
说明 M3"keyword 低估真 LLM"的说法要修正:keyword 没怎么冤枉真 LLM,
真正的扭曲在 MockLLM 侧。

### M3 baseline 公平复核(keyword vs LLM-judge)

| 配置 | keyword | LLM-judge | 差 |
|---|---|---|---|
| Coder 纯LLM | 0.30 | 0.31 | ~0 |
| Coder 原始码RAG | 0.16 | 0.15 | ~0 |
| Coder 图卡片 | 0.59 | 0.56 | −0.03 |
| Coder 全栈 | 0.75 | 0.74 | −0.01 |
| 通用 全栈 | 0.83 | 0.77 | −0.06 |
| **MockLLM 图卡片** | 0.81 | 0.70 | **−0.11** |
| **MockLLM 全栈** | 0.91 | 0.81 | **−0.10** |

**结论(修正并精确化 M3 caveat)**:
1. **keyword 对真 LLM 基本公平**(差 ≤0.06),对 **MockLLM 过度奖励 ~0.10**
   ——抽取式逐字倒卡片正中 keyword 的下怀。
2. 公平裁判下全栈排名:**MockLLM 0.81 > 通用 0.77 > Coder 0.74**。MockLLM 仍最高,
   但对真 LLM 的优势从表观 +0.16 缩到 +0.04~0.07。MockLLM 不全是"作弊":
   对"列出多个"类问题,完整倒出正确卡片就是高覆盖答案,真 LLM 反而精简/漏项。
3. **E19 的核心结论(通用>Coder)在公平裁判下依然成立**(0.77>0.74),稳健。
4. 此后**以 LLM-judge 为主指标**,keyword/Recall/MRR 为辅助硬信号。

## E21:PE 四维度系统优化(逐项增益 + 归因瓶颈)

底座=通用 Qwen2.5-14B + 全栈 RAG(M3 较优组合),baseline=朴素 prompt
(LLM-judge 0.77)。四维度各做"加 vs 不加"独立对照,主指标 LLM-judge。
prompt 模板项目无关(System/CoT 不含 werkzeug,Few-shot 用通用迷你项目),
完整方案见 PE_SOLUTION.md。

| 配置 | LLM-judge | Δ | 间接依赖 | 继承 | 反向依赖 |
|---|---|---|---|---|---|
| baseline(plain) | 0.77 | — | 0.40 | 0.89 | 0.69 |
| +System(角色+格式约束) | 0.78 | +0.01 | 0.60 | 0.89 | 0.65 |
| +Few-shot(通用 20 条) | 0.73 | **−0.04** | 0.70 | 0.76 | 0.67 |
| **+CoT** | **0.80** | **+0.03** | **0.70** | 0.89 | 0.68 |
| +All(Sys+FS+CoT) | 0.79 | +0.02 | 0.60 | 0.89 | 0.70 |
| +All+后处理 | 0.78 | +0.01 | 0.60 | 0.89 | 0.72 |

**结论(每项独立量化 + 归因,对口考核"逐项数据"+导师"适用边界"):**
1. **CoT 最有效(+0.03),增益几乎全来自间接依赖 0.40→0.70(+0.30)** ——
   多跳依赖是可被思维链引导的推理瓶颈;这是 PE 的主场。
2. **通用 Few-shot 是负结果(−0.04)**:项目无关示例与强 RAG 上下文**争夺注意力**,
   把继承从 0.89 拉到 0.76。诚实记录的负结果。
3. **System 轻微正**(格式/诚实约束);**后处理只微帮反向依赖漏列**(0.69→0.72)。
4. **最关键的归因**:**反向依赖高扇入几乎不被任何 PE 改善**(0.69→0.72)——
   它是**检索的锅**(目标依赖卡片没排进 top-k),PE 治不了;而**间接依赖被 CoT
   大幅改善**(推理的锅,PE 治得了)。**"PE 管推理瓶颈、RAG 管检索瓶颈"的分工**
   清晰成立,这是组合策略(M6 消融矩阵)的依据。
5. 推荐 PE 方案 = **System + CoT(0.80),不加通用 few-shot**。

> 判定:LLM-judge 为主(已校准,E20),keyword/Recall/MRR 辅助。生成温度 0,可复现。

5. 三手段分工至此完整:**RAG 管检索瓶颈、PE(CoT)管推理瓶颈、FT 在本任务上无独立
   增量(且有过拟合风险)** —— 直接喂给 M6 的六格消融矩阵。

## E26:few-shot 澄清对照(领域 vs 通用)

M4 发现"通用 few-shot 有害",但不等于所有 few-shot 都该丢。补领域内 few-shot
(21 条,从 flask/click 真实依赖 Q&A 取,与 werkzeug 隔离)对照:

| 配置(均含 PE System+CoT + RAG) | LLM-judge | hard |
|---|---|---|
| 无 few-shot(PE+RAG) | 0.80 | 0.65 |
| + 通用 few-shot(M4,玩具项目) | 0.79 | 0.63 |
| **+ 领域 few-shot(真实依赖 Q&A)** | **0.81** | **0.66** |

**结论:few-shot 的有效性取决于"示例与任务/上下文的相关性"。**
通用玩具示例与真实 RAG 上下文争注意力 → 有害(−0.01);领域内真实依赖示例
教对了推理/格式 → 有益(+0.01)。**精炼的最优方案 = 领域 few-shot + System + CoT + RAG = 0.81**。

## E27:第二项目泛化验证(rich,风格迥异)

## E32:第一层 统计 —— 15 轮重跑 + 95%CI + Welch t

每配置 15 轮(temp=0.7 生成,temp=0 判分),werkzeug 56 用例。

| 配置 | overall | σ | 95%CI | hard |
|---|---|---|---|---|
| RAG only(plain) | 0.7737 | 0.0123 | [0.767, 0.781] | 0.6142 |
| PE+RAG(System+CoT) | 0.7933 | 0.0166 | [0.784, 0.803] | 0.6393 |
| All(PE+RAG+FT) | 0.7955 | 0.0223 | [0.783, 0.808] | **0.5724** |

| PE+RAG+领域few-shot | 0.7994 | 0.0084 | [0.795, 0.804] | 0.6456 |

**关键对比(Welch t):**

| 对比 | Δ | p | 判定 |
|---|---|---|---|
| **PE+RAG − RAG only** | **+0.0197** | **0.0011** | ✅ **显著(真信号)** |
| CoT 在 hard 上 | +0.0251 | 0.0034 | ✅ 显著 |
| All(+FT) − PE+RAG(overall) | +0.0022 | 0.764 | ❌ 噪声内 |

| PE(System only) − RAG only | +0.0034 | 0.484 | ❌ 噪声内 |
| +领域 few-shot | +0.0061 | 0.220 | ❌ 噪声内 |

**裁决(部分修正 M8):**
1. **CoT 增益是真信号**(+0.02 overall p=0.001;hard +0.025 p=0.003)。M8 凭 3 轮 σ 把它判为
   "噪声内",是**统计功效不足的假阴性**——样本从 3→15,CoT 浮出显著。**这条 M8 软化被本层翻回。**
2. **System-prompt 单独 / 领域 few-shot:确认噪声内**(p=0.48 / 0.22),M8 结论维持。
   推论:PE 的增益**全部来自 CoT**,不来自 System Prompt 或 few-shot。
3. **FT:overall 无增量(p=0.76),且在 hard 上显著回归 −0.067(p<0.0001)**。FT 出局更硬:
   不是"分数持平所以看成本",而是"hard 上有统计显著的负作用"。(注:此为 hard 难度桶整体,
   与 M8 推翻的"indirect 子类崩到 0.20"是不同粒度——indirect 专项 n=18 未崩 仍成立,
   但 hard 桶整体 FT 确有显著拖累。两者不矛盾,已分别诚实记录。)

**对"谜题"的第一层回答:** "模型侧全在噪声"被证伪——**CoT 是小而真的信号**;真正在噪声内的是
System/few-shot/FT-overall。"RAG 压倒一切(+0.5)"依然成立,但"除 RAG 外全是噪声"是统计假阴性。

## E33:第二层 RAG 实现 —— 强度梯度(嫌疑最大的混淆变量)

假设(待排除):依赖图卡片由 tree-sitter 精确导出,可能把答案近乎逐字喂到模型嘴边,
使任务退化为"复述",从而让模型侧优化显得冗余。做法:prompt 文本层退化卡片的"边"
(`degrade_cards.py`,保持检索不变,只改卡片信息质量),四档重测 CoT 增益。

| RAG 强度 | overall(plain → +CoT) | ΔCoT | p | hard ΔCoT | p(hard) |
|---|---|---|---|---|---|
| (a) 完美卡片 | 0.774 → 0.793 | **+0.020** | **0.001** ✅ | **+0.025** | **0.003** ✅ |
| (b) 不完整卡片(随机删边) | 0.616 → 0.609 | −0.007 | 0.24 ❌ | −0.003 | 0.74 ❌ |
| (c) 带噪声卡片(注入错边) | 0.566 → 0.576 | +0.010 | 0.26 ❌ | +0.008 | 0.53 ❌ |
| (d) 仅原始代码(无卡片) | 0.177 → 0.173 | −0.005 | 0.35 ❌ | +0.002 | 0.74 ❌ |

**两个独立结论:**
1. **卡片确实是性能主轴**:退化卡片使 overall 从 0.77 雪崩到 0.62→0.57→0.18。
   "RAG 压倒一切"不是评测假象——把卡片信息删/污染掉,分数就垮,证明分数确由卡片信息支撑。
2. **CoT 增益不随 RAG 变弱而复活,反而消失**:CoT 仅在**完美卡片**下显著(+0.020/+0.025),
   一旦卡片不完整/带噪/缺失,CoT 增益立刻退回噪声内(p=0.24/0.26/0.35)。

**裁决(排除嫌疑最大的变量):** "强 RAG 抹平难度、使模型优化冗余"的假设**被证伪**——
若它成立,弱 RAG 下 CoT 应**变显著**;实测恰好相反,弱 RAG 下 CoT **更不显著**。
机理:**CoT 是"正确上下文之上的推理放大器",不是检索的替代品**——卡片对了,CoT 才有料可推
(把好上下文多榨 +0.02);卡片错了,没有正确信息可推理,瓶颈回到"信息可得性"(检索),
CoT 无能为力。→ 该混淆变量排除,"RAG 主导"是任务内禀属性,不是卡片喂答案的伪影。

## E34:第三层 评测 —— 细粒度裁判 vs 粗判定

假设(待排除):比例式裁判只数"实体命中",可能把"答案变好但没完全变对"的改善吃掉,
低估模型侧优化。做法:对**同一批 temp=0 答案**(锁死生成),用两套裁判打分对比 CoT 增益。
judge1=原比例式(命中数/要点数);judge2=细粒度多维(实体0-2+关系0-2+推理/清晰0-1,/5 归一)。

| 裁判 | overall CoT 增益 | p |
|---|---|---|
| judge1 比例式(实体命中) | +0.044 | 0.021 ✅ |
| **judge2 细粒度多维** | **+0.083** | 0.033 ✅ |

**裁决(部分确认该变量):** 细粒度裁判看到的 CoT 增益约为粗判定的 **2 倍**。CoT 改善了
"依赖关系方向正确、推理清晰、部分要点对"这些**比例式裁判数不到**的维度。
→ 粗判定**低估**(而非高估)CoT;"CoT 只有 +0.02"本身部分是评测地板效应,细看更大(但仍 ≪ RAG 的 +0.5)。
这进一步**反向 M8 的"CoT 在噪声内"**:不仅显著,真实效应还被主裁判压低了。
**局限:** 本层 n=3、temp=0,幅度为指示性;judge2 未做人工校准(judge1 M4 已校准 MAE0.022),
故 judge2 仅作"分辨率探针",不取代主指标。稳健结论是**方向**:细判定→CoT 增益更大,不更小。

## E35:第四层 模型 —— 跨模型确认(Coder-14B,不同家族)

前三层已解释现象,故本层按 M9 降为小规模确认:换**不同家族**的 Qwen2.5-Coder-14B

| 模型 | overall ΔCoT | p | hard ΔCoT | p |

**裁决:** CoT 的小而真增益在**不同家族模型上复现**(Coder 上甚至更大 +0.034)。

## 三条诚实结论(结论随对照翻转,最终以归因为准)

1. **〔回答 M11 主问题〕相似度检索单独用,远差于依赖检索**(−0.47 werkzeug / 类似 rich,p<1e-6,跨项目一致)。
   文献那个"相似度检索 +68~98%"是**代码补全**任务;本项目是**NL 依赖问答**,NL 问句对代码块语义相似度弱,
   而对结构化依赖卡片匹配强。**适用边界结论:依赖问答任务上,结构化依赖检索 > 相似度检索,与代码补全任务最优策略不同。**

2. **〔真正挖到的改进〕项目一直被一个欠调超参卡着:`rerank_candidates=12` 太浅。**
   提到 80 后(纯深度效应):werkzeug overall **0.793→0.936(+0.142,p<1e-6)**,反向依赖 **0.697→0.953**。
   机理(已逐例验证 REV-01):反向依赖的答案在"目标文件自己的依赖卡片"里(被依赖行),但浅 rerank 只看 12 个候选,
   目标卡常排在更深处被漏掉;深检索把它捞回来。**这是 M1-10 全程(都用 rerank_candidates=12)被低估的天花板。**

3. **〔自我纠错〕"显式融合分槽"是多余甚至有害的。**
   我一度以为"融合 3dep+2sim=0.920 是新最优",但 deep-mixed 控制显示:**只要给够深的候选池,
   cross-encoder reranker 自由挑 top-5(0.936)就比任何强制分槽(纯卡片 0.885 / 融合 0.920 / 纯相似 0.418)都好。**
   reranker 已经会"该多给卡片时给卡片、该带代码时带代码"——人为锁死比例反而打断它。

> **小样本警示**:dataflow(n=3)是唯一"相似代码必需、依赖卡片答不了"的类型(纯卡片 0.067 → 含相似代码 1.000),
> 但仅 3 例,只能作"存在这类问题"的提示,不作定量结论;indirect(n=5)上默认混合反而偏高(0.607),同属小样本。
> 稳健的是大类:reverse(13)、forward(11)、inheritance(12)、symbol(12)与 overall(56)。

**M11 行动项**:推荐配置改用 **rerank_candidates=80**(一行配置),werkzeug 0.94 / 反向依赖 0.95;
相似度检索不单列、由 reranker 在深池中自然纳入即可。

## E39:选择性检索 + 自评估(Repoformer 式,arXiv 2403.10059)

针对实测痛点("PE only < baseline,脱离可靠上下文模型会瞎猜")做自评估:答前先自评上下文是否足够,
不足则诚实输出"信息不足"而非编造(`configs/prompts/selfassess.yaml`)。两维度:准确率 + 幻觉(自信错误)率。
自信错误 = 未 abstain 且 score<0.5(自信地答错)。

| 条件/配置 | 准确率 | abstain率 | **自信错误(幻觉)率** |
|---|---|---|---|
| 无RAG · plain(纯瞎猜) | 0.202 | 0.001 | **0.790** |
| 无RAG · 普通CoT | 0.102 | 0.649 | 0.264 |
| **无RAG · 自评估** | 0.001 | 0.998 | **0.001** |
| 深RAG · 普通CoT(pe_cot_deep) | 0.936 | ~0 | — |
| **深RAG · 自评估** | 0.892 | 0.064 | 0.044 |

**裁决(价值在可信度,不在涨分——与预先声明一致):**
1. **核心价值:上下文不足时把"自信的错答案"换成"诚实的 abstain"**。无 RAG 下,plain 瞎猜→**79% 自信错误(幻觉)**;
   自评估→**0.1% 自信错误**(99.8% 诚实说"信息不足")。对代码工具,"我信息不够"远胜一个自信的错误依赖断言。
2. **代价:好上下文下会过度 abstain**。深 RAG 下自评估比普通 CoT **掉 ~0.044 准确率**(6.4% 把本可答的也 abstain 了)。
   说明 prompt 级自评估**校准偏保守**,不完美。
3. **总分不涨反略降**——但这正是该机制的设计取舍:**牺牲少量准确率换取幻觉率从 79%→0.1%**。
   若代码工具场景里"自信的错答案"代价高,这个取舍值得;若追求覆盖率则不值。**两面都如实呈现。**

> 局限与延伸:本阶段实现了自评估的 **abstain** 半边;"abstain→触发更深检索"的选择性检索全闭环留作 NEXT_STEPS。
> 校准偏保守(好上下文 6.4% 误abstain)可用阈值/few-shot 校准改善。

# M13续 微调负结果诊断:真结论 / 实验局限 / 代码bug?(E40)

> 背景:外部文献(FSE2025 RAG-or-FT、StackRepoQA)显示微调生成器在仓库级任务上通常是

> 不让一个可能由口径/局限/bug 导致的结果以"科学结论"身份留在交付里。**全部用已有数据 + 训练曲线,无需新 GPU。**

## 诊断1:口径核对(n=15,Welch t)—— 直接化解矛盾

| 对比 | Δ | p | 含义 |
|---|---|---|---|
| **FT only − baseline(都无RAG,overall)** | **+0.043** | **<0.0001** | ✅ **FT 本身显著为正,不是有害** |
| **FT only − baseline(hard)** | **+0.111** | **<0.0001** | ✅ FT 单独在 hard 上也显著为正 |

## 诊断2:训练健康度自检(排除 bug)

- **eval_loss 单调降不回升**:0.0336→0.0142→0.0115→0.0106→0.0093→0.0092(step90/93),无过拟合拐点。

- **adapter 正确加载已被诊断1证明**:FT only 用 model=dep(base+adapter);若 adapter 没加载,FT only 应 = baseline;
  实测在**完全留出的 werkzeug 上**显著超过 baseline(+0.043,p<1e-4)——这比"训练集内样本变好"是**更强的证据**

- 538 条(485 train/53 val)低于文献所述 FT 显效门槛(数千~亿级 token)。
- **但本项目无需用数据规模解释负结果**——因为**根本没有负结果**:FT 在 538 条下已显著为正。

  **"未充分探索大数据量下 FT 的上限,但在本数据量下 FT 已为正、且比 RAG 弱一个数量级"。**

## 最终结论(分级 = A,精确表述)

> **A 类(口径问题,与文献一致,精确化表述):微调生成器对依赖问答有真实正向效果**
> (overall +0.043 / hard +0.111 vs 无上下文基线,p<1e-4,与 FSE2025/StackRepoQA 一致);
> **但比 RAG 弱一个数量级(+0.04 vs +0.72),且在已有 RAG+CoT 的最优配置上冗余(无额外增量)。**
> 因此 FT **不进推荐方案——理由是"被 RAG 主导且冗余",不是"FT 本质无效"。** 训练健康、adapter 正常、无 bug。

**已作废的旧表述**(全项目回填):"FT 性价比为负"→改为"FT 正提升但弱于&冗余于 RAG";
"FT 伤 hard −0.067"→澄清为"FT *单独*提升 hard(+0.111);仅在叠加到 PE+RAG 时 hard 边际 −0.009(深度修正后近零,旧 −0.067 系浅检索叠加伪影)";
"微调出局"→保留(不进方案)但理由精确化。

# M14 微调检索器(对比学习 embedding)—— 补上唯一空白,对标 RLCoder(E41)

> 项目唯一实打实的空白:此前只微调了生成器(Qwen-14B LoRA),从未微调检索器。文献(RLCoder ICSE2025)

> (InfoNCE/MultipleNegativesRankingLoss),回答:**微调检索器 vs 微调生成器,谁是更大杠杆?**
> 红线:159 个 (查询,正卡片,难负) 三元组**只来自 flask/click/jinja2/requests,werkzeug 零参与**(已核验 0 泄漏)。

## 训练健康度
- 对比损失单调降:2.26 → 0.21(3 epoch,本地 GPU 4.8s)。基座 bge-small-zh-v1.5。负样本:in-batch 随机 + 同目录难负。

## 三层对照(统一口径,只换"检索器"一个变量)

**① 隔离 embedding 效果(纯向量检索,无 BM25/rerank):Recall@5 / MRR**
| 项目 | 现成 bge | 微调 bge | 变化 |
|---|---|---|---|
| werkzeug | 0.679 / 0.495 | 0.491 / 0.438 | **overall ↓**;easy 0.89→**0.22**、hard 0.50→**0.73** |
| rich(跨项目) | 0.615 / 0.637 | 0.519 / 0.479 | **overall ↓**;easy 1.00→**0.17**、hard 0.50→**0.89** |

→ **微调 embedding 过度专化**:在依赖查询(hard)上变好,但 easy(符号定位)灾难性崩塌,整体 Recall 下降。**跨项目一致。**
机理:159 条窄分布(全是依赖查询)+ 小模型,过拟合到"依赖卡片匹配",丢失通用语义匹配。

**② 全管线 Recall@5(hybrid + BM25 + cross-encoder rerank=80)**
| | 现成 bge | 微调 bge |
|---|---|---|
| werkzeug Recall@5 | 0.9464 | **0.9464(完全持平)** |

→ **embedding 的过度专化被全管线冲掉**:BM25 补回 easy(精确名匹配)、深 rerank 主导最终排序。**embedding 不是瓶颈。**

**③ 端到端 LLM-judge(n=15,rerank=80)**
| | overall | p |
|---|---|---|
| 现成 bge(werkzeug_pe_cot_deep) | 0.9356 | — |
| 微调 bge(werkzeug_pe_cot_deep_ft) | **0.9150** | **−0.021,p=0.003 显著(略降)** |

→ 过度专化的轻微泄漏使端到端**略降**(file 级 Recall 持平,但喂给 LLM 的具体块/序略差)。**微调检索器没赢,还略输。**

## 两个微调杠杆横向对比(本项目最终定论)

| 杠杆 | 效果 | 性质 |
|---|---|---|
| 微调生成器(M5/E40) | +0.04 vs baseline(真实)、最优配置上冗余 | 边际,被 RAG 主导 |
| **微调检索器 embedding(M14)** | 纯向量过度专化(overall↓)、全管线冲掉、端到端 **−0.021** | **不是杠杆(略负)** |
| 非学习:结构(依赖卡片) | baseline 0.19→0.77 | 数量级杠杆 |
| 非学习:检索深度(rerank 12→80) | +0.14 | 数量级杠杆 |

## 结论(诚实,边界清晰)

> **边界结论:对依赖问答,检索侧已被"静态分析依赖卡片 + 现成 embedding + BM25 + 深度 cross-encoder rerank"近乎饱和
> (Recall@5 0.95);最小可行版的学习式 embedding 微调过度专化、打不过它,端到端还略降。**
> 回答核心问题:**微调检索器 vs 微调生成器——两个学习式模型侧微调都是边际/负,真正的杠杆是非学习的结构 + 检索深度。**
> 这与 RLCoder 的"检索侧是更大杠杆"在本任务上**不一致**——但口径不同:RLCoder 微调的是检索器在通用补全任务上,
> 且用 RL + 大数据;本项目是最小可行对比学习 + 159 条窄数据 + 已近饱和的强 baseline。
>
> **诚实边界(防过度推论)**:这是**最小可行版**(159 条窄分布 + bge-small),过度专化部分源于数据小而窄;
> 更大/更均衡的数据或 RL 式训练**可能**改善,未探索。但鉴于全管线 Recall 已 0.95、深 rerank 已主导,
> **embedding 微调的上行空间被强 baseline 压得很小**——这本身就是"规则式/现成检索已接近上限"的有力证据。

## E43:领域自适应预训练 DAPT(无监督继续预训练 + 遗忘监控)

### 概念(与 SFT 区分,见 DAPT_NOTES.md)
SFT=喂"问题→答案"教答题;DAPT=拿领域代码做 next-token 让模型**沉浸领域分布**(无标签)。核心问题:
**让模型大量"读"同领域库代码,能否学会"谁 import 谁"这类跨文件结构事实?**

### 轻量化 + 隔离
LoRA-based 继续预训练,硬上限 **200 步**;语料 722 段来自 click/jinja2/requests(**排除 flask** 防泄漏 + 段级过滤,
**0 werkzeug**,训练启动再 assert)。遗忘监控:12 题 HumanEval 式探针 pass@1(与依赖问答无关)。

### 结果(n=15 base 裁判 + 探针)

| 指标 | base | DAPT 后 |
|---|---|---|
| 依赖问答 FT-only(无 RAG) | 0.185(baseline) | **0.173±0.011** |
| 通用代码 pass@1(遗忘探针) | 1.000(12/12) | 1.000(12/12) |

- **DAPT 在依赖问答上无增益**:DAPT 0.173 vs baseline 0.185,**Δ=−0.012,p=0.099 噪声内**(甚至略低)。
- **轻量 DAPT 未见可测遗忘**:探针 pass@1 前后都 12/12(注:本探针偏易、有天花板效应;FSE 报告完整大规模 DAPT 可掉 HumanEval ~24%,
  本最小可行版步数少故遗忘不明显——诚实标注未充分探索大规模 DAPT 的遗忘代价)。

**裁决(最深的论断佐证):依赖关系不在代码文本的统计分布里。**
让模型大量"阅读"领域代码(DAPT)不能学会跨文件依赖——因为依赖是**"算"出来的结构事实(谁 import 谁,由静态分析确定),
不是"读"出来的统计规律**。这正是"必须用静态分析依赖图卡片检索、而非靠模型记忆/沉浸"的根本原因。

- 渠道B:别名继承(INH-07/08)、传递依赖(IND-01/05)**人工读真实源码核验**。

## E68(M38-40)识破≠精确补全:鸿沟是"信息可得性"边界,非推理/prompt 缺陷(新且硬)
钻 M36 暴露的"识破 93% vs 精确补全 1%"。分开测两确定性指标:detect(点出存在脏依赖)vs complete(具体补出脏模块名+机制)。

**机理实验(determinable 子集 n=58,补全才可能)**:humble(仅卡片)complete **10%** → **humble_source(卡片+真实源码片段)90%**(Δ+0.805,**p=2.0e-25**)→ humble_prompt(显式要求点出哪个模块+机制)74%(Δ+0.644,p=3.0e-14)。
**裁定(新且硬):识破-补全鸿沟是【信息可得性】边界,非模型推理差也非 prompt 没引导。** 静态卡片是扁平列表、缺脏依赖的结构(哪个是条件/动态),模型无法从扁平卡片凭空补出具体缺失模块(10%);**把真实源码喂进上下文→精确补全跳到 90%**(证明不是推理失败);显式 prompt 能调动部分参数知识(74%)但不如喂源码。
**对 M36 的诚实校准**:M36"humble 只认性质不定位机制(条件 1%)"在大样本坐实(conditional complete 3%),但**prompt 把 determinable 补全抬到 74%、喂源码抬到 90%**——局限真实但**prompt 可部分解、源码可基本解**,精确定位为"信息可得性"而非"模型无能"。
**产品姿态(比'用humble'深一层)**:humble 让模型**警觉**(识破 99%)但不能让它**全知**(仅卡片补全 10%)。正确架构=**模型识破脏依赖→触发针对性再分析(把真实源码/动态分析结果喂回上下文)→模型据此精确补全(90%)或转人工确认**,而非让模型凭参数知识自补(10%)、也非只靠 prompt 硬调(74% 仍漏 26%)。VALUE_FOR_REPOMIND §2c、INSIGHT。
