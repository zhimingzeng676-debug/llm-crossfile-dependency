> 每条标:哪天发现、是什么、为什么可能有趣、是否值得做成超额(待补完主线后评估)。

## 1. 〔M64,PE 后处理〕过滤(filter)在本管线 Δ=0,因为反幻觉该在 System Prompt 层而非后处理层
- **现象**:反幻觉接地过滤(剔除答案里检索上下文没有的文件名)在 werkzeug 56 例上剔除 0 项、Δrecall=0——模型几乎不报未检索的文件名。
- **为什么有趣**:说明 System Prompt 的"只依据上下文、不编造文件名"约束已在**生成阶段(预防)**杜绝了文件名幻觉,使**后处理阶段(事后)**的过滤冗余。这是一个"预防 > 事后纠正"的小机理。
- **是否值得做成超额**:温和。可做一个对照(去掉 System Prompt 反幻觉约束→看 filter 是否就有活干了),量化"System Prompt 预防 vs 后处理纠正"的分工。✅ **已做成超额(M69)**:System on/off 量化,预防减幻觉 66%>纠正,见 `PE_SOLUTION §五`。

## 2. 〔M64〕det-recall 下补全增益(+0.047)显著大于 LLM-judge(+0.01)
- **现象**:同一补全操作,判分无关 det-recall 测出 +0.047,LLM-judge 只测出 +0.01。
- **为什么有趣**:LLM 裁判对"补回漏列项"不敏感(可能因裁判本身对完整性宽容/部分分),而确定性 keyword 指标直接量到。这是"裁判指标选择影响能力测量"的一个小例证,呼应项目主线"可信度靠判分无关锚"。
- **是否值得**:✅ **已做成超额(M70)**:det vs judge 双向分歧对照,互补非取代,见 `JUDGE_INDEPENDENT_VALIDATION` M70 节。

## 3. 〔M65,RAG embedding〕cross-encoder rerank 完全掩盖了 3× 的 embedding 质量差异
- **现象**:纯语义层 bge-small-zh(R@5 0.32)vs bge-small-en(R@5 0.93)差 ~3×;但全栈(+BM25+rerank)即便用最差的 zh 也达 R@5 0.79——rerank 把弱 embedding 的排序补回。
- **为什么有趣**:量化了"深 rerank 对 embedding 选型的鲁棒性"——一个强 rerank 让上游召回器的质量差异在端到端上几乎消失。这解释了为什么项目一直用次优中文 embedding 却没在最终分上暴露。
- **是否值得**:✅ **已做成超额(M68)**——rerank×embedding 2×N 量化:rerank 把 3× embedding 差异压缩 **11.3×**;工程边界见 `EVALUATION_REPORT.md §3c` / `VALUE §9`。

## 4. 〔M66,长上下文〕位置效应是 recency(末尾最好)而非经典 lost-in-the-middle(中间最差)
- **现象**:卡片在 44k 上下文里,末尾 0.910 > 中间 0.753 > 开头 0.684——单调 recency(越贴问题越好),非 U 形(中间最差)。
- **为什么有趣**:经典 LITM 文献常见 U 形;本任务(依赖卡片检索)是单调 recency。可能因任务是"找一条权威信息"而非"综合多处",或 Coder-14B 的位置偏好不同。
- **是否值得**:✅ **已做成超额(M71)**:5 位置细扫+跨模型,确认 recency 非 LITM、幅度模型依赖,见 `DIAGNOSIS` M71 节。
