## 1. 项目选择:CPython
| 项 | 值 |
|---|---|
| 项目 | **CPython**(python/cpython,shallow clone) |
| 为何选它 | 业界标杆、Python+C 典型巨型混合、跨语言依赖真实大量、可处理(不像完整内核会烧光时间) |
| 数据隔离 | CPython 本体(解释器 + stdlib + C 核心),与此前所有测试库(werkzeug/flask/.../Go/Java/C 项目)**零重叠** |
| 规模 | **3408 源文件**(Lib/ 2006 Python + Modules/Objects/Python/Include 1028 C/H),vs werkzeug 52 = **~65× 跳跃** |
| 语言构成 | Python(stdlib)+ C(解释器核心、C 扩展模块) |
| 跨语言边界 | Python 文件 `import _X` → C 扩展模块 `Modules/_X.c`(含 `PyInit__X`),如 `Lib/json/decoder.py → Modules/_json.c` |

## 6. 最终定位:【依然成立】

**核心论断在工业级规模(65× 文件 / 14× 卡片池)上【依然成立】**,数据支撑:
1. **结构化表示仍是数量级杠杆**:full−baseline **+0.894(Coder,p=1.5e-24)**,baseline 近零。
2. **检索没成为瓶颈**:1456 大池下 `rerank=80` 仍捕获 **97.5%** 正确卡片、rerank 后 95% 进 top-8——**正确卡片没有淹没**,无需检索升级即成立。
3. **跨语言扩展到工业级**:可静态抽取的 Python↔C 扩展边 **+1.00**(完美),复现并扩展 M20"可抽取→成立"原则。

**诚实边界(规模维度)**:
- 本实验测的是 **14× 卡片池(1456)**,`rerank=80` 足够;**未触到硬规模边界**。
- **方向性信号**:py_forward 在 @8 仅 0.50(@20=0.88)——卡片词面竞争随池增大而加剧。
  **外推**:在远大于此的规模(100×+,上万卡片)下,检索深度/**分层(层级)索引**可能成为必需,
  但本实验**未证实**这是硬边界,如实标注为"待更大规模验证"。
- **真正的边界仍是"结构能否静态计算"(同 M20),非"池太大"**:条件 import、`importlib` 动态加载、
  构建系统隐式链接、Argument Clinic 生成代码、vendored 依赖会让解析抓不全比例随规模上升——这是规模带来的真实挑战。
- 金标准可信度:大项目无法人工核验全部,抽样人工核验(abc/atexit/base64 等)+ ast/regex 独立抽取,可信度随规模略降,已声明。
