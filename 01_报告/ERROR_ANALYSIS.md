# ERROR_ANALYSIS — 冠军配置的逐用例误差剖析

> 对象:`best_stack_norerank`(答案分 0.87 / 检索命中 0.93,41 条用例中 35 条满分)。
> 取数工具:`python scripts/list_errors.py`(任何配置的结果 JSON 都能剖)。
> 方法:对每条非满分用例,用双指标 + 检索块列表 + 答案原文,把失败定位到
> 链路的具体环节;再归纳错误类型分布。判定规则已在本阶段审计收紧
> (见 WORKLOG 决策 12),因此下面没有"判定误判"型错误。

## 错误类型分布(6 条非满分用例)

| 类型 | 用例 | 数量 | 一句话 |
|---|---|---|---|
| A. 检索漏召回(真难例) | HI-07 | 1 | 该召回的块不在 top-k 里 |
| B. 检索"命中"但块不含答案 | SE-04, CF-05 | 2 | 文件对了、块错了 —— 暴露指标粒度盲点 |
| C. top-k 组成被同源块稀释 | CC-02 | 1 | 答案块在,但被三张图卡片挤占名额 |
| D. 生成侧(MockLLM 局限) | CC-07, HI-08 | 2 | 检索全对,mock 的意图路由/选行失误 |
| E. 用例本身有歧义 | — | 0 | 判定审计后清零 |

**先说一个比单条错误更重要的发现(来自 B 类):`retrieval_hit` 指标按
"文件前缀"判命中,SE-04/CF-05 里它是 1.0,但检回来的是该文件中不含答案的块。
也就是说 0.93 的检索命中率有约 2/41 的"虚高"。** 改进方向:把
expected_sources 升级到块粒度(或加"答案串是否出现在检索文本里"的第三指标),
已记入 NEXT_STEPS。诚实的评测先要会怀疑自己的指标。

## 逐用例剖析

### HI-07「优惠券模块修过什么 bug?」 — 0 分,检索命中 0(A 类)
- 现象:top3 = 优惠券的 **feature** commit + 无关的手续费 fix commit + coupons 模块头,
  目标 bugfix commit(4e9f2c8"抵扣不能超过支付金额")没进来。
- 根因:问题用词"修过 bug",commit 用词"fix: ...不能超过...负金额"。
  二者词面零重合,语义关联也弱(bge 同样没救回来 —— 这条在全部 10+ 配置下都是 0)。
  本质:**"bug 修复"是 commit 的元属性,不是 message 的内容**,靠内容检索找元属性,
  方向就不对。
- 改进:解析 commit message 的 `feat:/fix:/refactor:` 前缀为结构化元数据,
  历史类查询先按类型过滤再检索(成本极低,NEXT_STEPS A 级)。

### SE-04「多大金额算大额交易?」 — 0 分,检索命中 1(B 类)
- 现象:top3 含 risk.py 的 **assess_risk 函数块**(检索命中=1 据此成立),
  但 `LARGE_AMOUNT_THRESHOLD = 500_000` 在**模块头块**里,没被检回。
  答案引了 commit 和风险等级说明,就是没有数值。
- 根因:function 切块的结构性常量盲区(E4 已发现),叠加指标的文件粒度盲点
  (见上)。这条在 baseline(fixed 切块)下是满分 —— 切块策略的 trade-off 实锤。
- 改进:① 模块头块拆细(每个常量+它的注释一块);② 图卡片补"常量卡"
  (名字/值/所在文件,tree-sitter 解析赋值语句即可)。

### CF-05「MAX_DISCOUNT_CENTS 的值是多少?」 — 0 分,检索命中 1(B 类)
- 与 SE-04 同根因:值的定义行在模块头块,检回来的是 commit(提到常量名但没有值)
  和**使用**该常量的函数块。常量盲区的第二个实例,B 类不是孤例。
- 附注:这条本来还是"注释与代码不一致"陷阱(注释写 50 元,代码 3000 分),
  冠军配置根本没走到"信注释还是信代码"那一步就在检索侧输了。
  陷阱的真正考验要等常量盲区修复 + 真 LLM 接入。

### CC-02「process_payment 调用了哪些函数?」 — 0.33 分,检索命中 1(C 类)
- 现象:top3 全是图卡片(assess_risk / process_payment / validate_amount 的卡),
  process_payment 自己的卡片在第 2 位 —— 它的"它调用的仓库内函数: validate_card,
  validate_amount, assess_risk, calculate_fee, log_event"一行就是完整答案,
  但 MockLLM 的行打分被三张卡的"被谁调用"行(都含 process_payment,词面重合极高)
  淹没,答案只凑出 assess_risk 一个。
- 根因:**同源块互相稀释**:图卡片让"提到 process_payment 的块"数量翻倍,
  反而把"属于 process_payment 的块"挤出注意力。chunk_function/semantic 配置下
  这条是满分(直接检到函数代码块)。
- 改进:检索结果多样性约束(同一函数的卡片+代码块去重/合并);
  或 mock 选行时按"卡片归属主体==问题主体"加权。真 LLM 大概率不受此影响
  (它能读懂卡片归属),标注待复核。

### CC-07「assess_risk 定义在哪个文件?它被谁调用?」 — 0.5 分,检索命中 1(D 类)
- 现象:检索三连中(assess_risk 卡、is_blacklisted 卡、risk.py 文件卡),
  信息全齐。但答案是"其中发起的函数/构造调用有: import" —— 胡话。
- 根因:纯生成侧。问题含"调用"触发 mock 的调用提取模式,在**文件卡片**上
  用 `名字(` 正则抓到了"import"这种假调用;复合问题(在哪 + 被谁调)
  也超出了 mock 单意图路由的能力。
- 处置:**不再给 mock 堆第 6 条启发式**(边际价值已耗尽,规则互相打架的
  风险在涨)。标注为"真 LLM 复核名单"第一名 —— 检索侧已无可改。

### HI-08「2026 年 5 月仓库里发生了哪些改动?」 — 0 分,检索命中 1(D 类)
- 现象:检回了 5 月的 commit(5fa03d2),但答案全是"提交 hash 作者 日期"行,
  没有一条"提交说明"行 —— 判定要的"优惠券/checkout/废弃"都在说明里。
- 根因:生成侧选行 —— 问题里的"2026 年 5 月"与日期行(2026-05-26)词面重合
  极高,日期行把说明行全部挤出名额。另外这本质是**聚合类问题**(理想答案要
  汇总 4 条 commit),top_k=3 + 抽取式生成的结构上限。
- 改进:聚合类问题适合"先按时间过滤元数据再汇总"的工具路线(和 HI-07 同方向),
  不适合硬靠相似度检索。

## 结论

1. 冠军配置的 6 条失分里,**真正的检索失败只有 1 条**(HI-07),
   其余是切块结构(2)、结果多样性(1)、mock 生成局限(2)。
   检索主链路已经相当强,下一步收益在**元数据/结构化路线**
   (commit 类型过滤、常量卡、块粒度命中指标),不在继续换更大的 embedding。
2. **需真 LLM 复核的结论清单**(mock 可能掩盖或制造的差异):
   CC-07/HI-08 的生成侧失败、CF-05 的注释-代码矛盾行为、
   E6 的 PE 全部结论、CC-02 的卡片稀释是否真影响生成。
3. 评测自身的改进项:retrieval_hit 升级到块粒度(0.93 中约 2 条虚高)。
