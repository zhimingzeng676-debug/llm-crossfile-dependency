## arXiv 2505.15179 —— 代码任务上检索策略的工业实证(M11 动机来源)

- **文献发现**:在代码任务上,**相似度检索**(retrieve 用法/语义相似的代码片段)收益很大
  (报告 EM +68%~98%),而**纯依赖检索**(RAG_Dependency,只检索结构化 import/调用依赖)
  收益小得多(仅 +8.9%)。
- **为何引它**:本项目的检索重心几乎全押在"依赖图卡片"(结构化 import/继承关系),
  按此文献可能**轻视了相似度检索这个更大的杠杆**——这是一个值得查的潜在盲区。
- **关键任务差异(避免误读)**:文献那个 68-98% 是在**代码补全**任务上测的
  (query = 当前代码上下文,retrieve 相似代码作 in-context 示例);
  本项目是**跨文件依赖问答**(query = 自然语言问题,answer = 结构性事实"谁依赖谁")。
  两个任务的检索最优策略不一定相同。

- **本项目验证结论(M11,EXPERIMENTS E37,n=15+95%CI,werkzeug+rich 双项目)**:
  1. **文献结论不迁移**:相似度检索**单独用**在依赖问答上远差于依赖检索
     (werkzeug 0.42 vs 0.89;rich 0.27 vs 0.91,p<1e-6,跨项目一致)。
     适用边界:**依赖问答任务上结构化依赖检索 > 相似度检索,与代码补全任务最优策略相反。**
     机理:NL 问句对代码块语义相似度弱(自然语言 vs 代码),对结构化依赖卡片匹配强。
  2. **意外收获(更有价值)**:验证过程中用 deep-mixed 对照发现,项目此前的检索一直被一个欠调超参
     `rerank_candidates=12` 卡着——提到 80 后 werkzeug 0.79→0.94、反向依赖 0.70→0.95(纯深度效应 +0.142,p<1e-6)。
  3. **自我纠错**:显式"融合分槽"多余——cross-encoder reranker 在足够深的候选池里自由挑,
     已自动做对依赖卡片与相似代码的混合,人为锁比例反而更差。

> 教训:引文献是为找盲区,不是为结论背书。文献指的方向(相似度检索)在本任务上**没用**,
> 但**沿着这个方向做对照,意外暴露了自己系统里一个真正的瓶颈(检索深度)**——
> 这正是"开放验证 + 控制变量"的价值:连否定结论的实验也会顺带挖出真问题。

## Repoformer(arXiv 2403.10059)—— 选择性检索 / 自评估(M12 任务二动机)

- **文献思想**:训练模型**自评估**"这次检索是否有必要/是否会帮上忙",对不需要检索的情况跳过检索,
  避免无谓甚至有害的检索(selective RAG)。核心是让模型**知道自己何时缺信息**。
- **迁移到本项目的痛点**:M10/12 实测的 "PE only < baseline"——脱离可靠上下文时,严格 grounding 的
  CoT prompt 会让模型**瞎猜**(或抑制后乱答)。借 Repoformer 思路做一个**答前自评估 + 选择性 abstain**:
  模型先判上下文是否足够,不足则**诚实输出"信息不足"而非编造**(`configs/prompts/selfassess.yaml`)。
- **本项目验证(M12,E39)**:对照 普通 prompt vs 自评估 prompt,在 深RAG / 无RAG 两条件下测
  **准确率 + 幻觉(自信错误)率**两个维度。**关键视角(预先声明)**:自评估的价值可能不在涨总分,
  而在**信息不足时把"自信的错答案"换成"诚实的 abstain"**——这本身是代码工具可信度的核心指标,
  即使总分不涨也要如实呈现。结论见 E39 / EVALUATION_REPORT。

## PE 效果上限(arXiv 2602.13890)—— 给"PE 只涨 2%"一个外部参照

- **文献结论**:在代码任务上系统评估 prompt engineering 的天花板,实证 PE 带来的提升上限在 **~6%** 量级。
- **本项目对标**:我们实测 CoT 在强 RAG 之上的边际 +0.02~0.03(System-prompt/few-shot 在噪声内)。
  **这与文献一致**——PE 在代码任务上本就是个位数边际效应。**"PE 只涨 2% 是不是 PE 太弱"是个伪问题**:
  不是我们 PE 做得差,是 PE 这条路在代码理解任务上**天花板本来就低**(与 RAG 的 +0.5~0.7 不在一个量级)。
  这正面支撑了核心论断:模型侧优化(含 PE)是边际的,RAG 才是数量级杠杆。

## 图谱式仓库理解(REPOGRAPH 类)—— 结构化图谱检索的同路印证

- **文献思路**:把代码仓库构造成图谱(函数/文件/依赖关系为节点边),供检索与多跳推理,提升仓库级代码任务。
- **本项目关系**:我们的"依赖图卡片"= 把调用/导入/继承图谱**文本化**成可检索文档(GraphRAG 思路的轻量实现)。
  **同思路、独立实证**:本项目用对照证明了"原始代码块 RAG(0.16)< 纯 LLM(0.31)< 图卡片 RAG(0.77→0.94)"——
  即**结构化图谱检索对依赖任务的碾压性优势**,且量化了 naive 代码 RAG 的有害性(这一点是独立贡献)。

> 对标总结:本项目结论与前沿在三处一致(PE 上限低、图谱检索是正解、自评估有可信度价值),
> 一处给出独立新边界(相似度检索不迁移到 NL 依赖问答)。**对标是为了定位自己,不是为结论找靠山。**

## 微调 vs RAG 在仓库级任务(FSE2025 RAG-or-FT / StackRepoQA / RLCoder)—— M13续 FT 诊断对标

- **FSE2025 "RAG or Fine-tuning"**:在仓库级代码任务上系统对比 RAG vs 微调生成器,结论是
  **微调通常带来正提升、但弱于 RAG**;RAG 在需要"外部/跨文件信息"的任务上优势明显。
- **StackRepoQA**:仓库级代码问答基准,印证依赖/跨文件类问题对**检索**的强依赖。
- **RLCoder**:用强化学习训练检索器(而非生成器)以提升仓库级补全——**说明"微调"在该领域更有效的用法
  往往是微调检索器/嵌入,而非微调生成器**。
- **本项目对标(E40)**:我们的诊断与这条线**高度一致**——FT 生成器在依赖问答上**正提升**
  (overall +0.04 / hard +0.11 vs 无上下文基线,p<1e-4)、但**弱于 RAG 一个数量级**(+0.04 vs +0.72),
  在最优 RAG+CoT 上冗余。**关键澄清**:本项目微调的是**生成器**且严守目标项目零参与训练;
  文献中"微调有效"多指微调**检索器**或喂**域内数据**——口径不同,结论不矛盾。
  因此本项目 FT 结论精确为:**"不喂目标项目数据、微调生成器做依赖问答,正提升但弱于&冗余于 RAG,故不进方案"**,
  而非"微调本质无效"。

> 诊断方法学:面对"自己测到的负结果与文献相悖",先做**口径核对**(FT only vs baseline)而非急着下结论——
> 一查发现根本不是"FT<baseline"(那才是真负),而是"强 RAG 之上边际为零",与文献完全一致。
> **很多'惊人的反直觉结论'其实是口径错位;诚实研究的第一反应应是'我是不是和文献在比不同的东西'。**

## RLCoder(ICSE2025)深化 —— M14 微调检索器实证

- **文献**:RLCoder 用强化学习训练**检索器**以提升仓库级代码补全,主张检索侧微调是更有效、更可泛化的杠杆。
- **本项目 M14 实证(E41 / RETRIEVER_FINETUNE.md)**:补上"从未微调检索器"这唯一空白——对比学习微调 bge-small embedding
  (159 三元组,werkzeug 零参与)。结果:**纯向量过度专化(easy 崩 hard 升,overall↓,跨项目一致)、全管线被 BM25+深rerank 冲掉
  (Recall 持平)、端到端 LLM-judge 略负(−0.021,p=0.003)**。
- **与 RLCoder 不一致但不矛盾(口径不同)**:RLCoder 是 RL + 大数据 + 通用补全任务;本项目是最小可行对比学习 + 159 条窄数据 +
  已近饱和(Recall 0.95)的强检索 baseline(静态卡片 + BM25 + 深 rerank)。**在本任务的强 baseline 上,小数据 embedding 微调没有上行空间。**
- **本项目最终定论**:**两个学习式模型侧微调(生成器 +0.04 冗余、检索器 −0.02 过度专化)都不是杠杆;
  真正的杠杆是非学习的结构(依赖卡片 +0.5)+ 检索深度(+0.14)。** 这是对"是否该投入学习式微调"的诚实回答:在本任务上不值得。

## 领域自适应预训练 DAPT(Gururangan ACL2020 "Don't Stop Pretraining" / FSE 遗忘报告)—— M15 E43

- **文献**:DAPT(领域继续预训练)在很多 NLP/代码任务上有效;但 FSE 等报告也指出**完整大规模 DAPT 会灾难性遗忘**
  (HumanEval 可掉 ~24%),是"无监督继续预训练"的已知代价。
- **本项目验证(E43,轻量 LoRA-DAPT 200 步,click/jinja2/requests,0 werkzeug)**:DAPT 在跨文件依赖问答上**无增益**
  (0.173 vs baseline 0.185,p=0.099);轻量版未见可测遗忘(探针 12/12,步数太少)。
- **独立机理论断**:DAPT 无增益**不是没训够**,而是**任务性质**——依赖是"算"出来的结构事实,不在代码文本的 next-token 统计分布里,
  所以"阅读"领域代码学不到它。这与"必须静态分析检索"的核心论断一脉相承。**与文献不矛盾**:文献的 DAPT 有效任务是"读得到统计规律"的;
  本任务恰是"读不到、必须算"的——给出 DAPT 适用边界的一个反例。

## 图 RAG / GraphRAG 对标(M19 E46)
- **REPOGRAPH / GraphRAG 类**:把仓库构造成图谱供检索/推理。本项目现状(GRAPH_REPRESENTATION.md)是其简化版——
  "后台用图算、文本喂模型"的 **1-hop 节点邻接卡片文本投影**(非真结构图喂模型/非子图检索/非查询时遍历)。
- **本项目独立实测的图表示梯度结论(E46)**:在去循环金标准+双独立裁判下,**图形态"按问题所需物化"**:
  直接依赖问题 1-hop 即天花板、图形态无所谓;**多跳/间接依赖必须把传递闭包物化(B2)**才解(0.40→0.99,+0.59 p=0.001)。
  → 对"GraphRAG 是否值得上复杂图结构"给出**任务相关的边界**:收益集中在多跳查询轴,直接查询轴收益为零。
  与 GraphRAG 文献"图结构对多跳问答有益"方向一致,并独立量化了"益在哪类、益多少"。

## 跨语言代码理解(M20 E47)
- **多语言代码依赖/检索**:业界代码理解多聚焦单语言(尤其 Python/Java)。本项目独立验证:**"结构化依赖表示是数量级杠杆"跨 Go/Java/C 通用**(强裁判 +0.60~0.93,p<1e-6),不是 Python import 机制特性。
- **C 的特殊性**:C 因宏 + 跨编译单元隐式链接,依赖结构最难静态抽取——本项目实测**可抽取的 #include 依赖上结构化照样碾压**,把边界精确定位到"不可静态计算的依赖类型"(宏/隐式链接),而非"C 语言本身"。这对"代码 RAG 是否跨语言"给出有边界的肯定答案。

## 仓库级 / 大规模代码理解(M21 E48)
- **仓库级代码理解**(RepoBench/CrossCodeEval/RepoCoder 等)关注大库下的检索与补全。本项目独立验证:**"结构化依赖表示是数量级杠杆"在工业级规模(CPython 3408文件)仍成立**(+0.89,p=1.5e-24),且**检索在14×卡片池下未成瓶颈**(rerank=80捕获97.5% gold)。
- **规模维度的边界**:把"大库检索"问题精确定位——14×规模 rerank=80 足够;外推100×+或需分层索引(未证实);真正边界是不可静态抽取的依赖(动态/隐式),非池大小。对"大项目是否需要层级化图检索"给出有数据的初步判断:14×尚不需要。

## REPOGRAPH / 图结构RAG 深化对比(M22 E49)
- **REPOGRAPH / GraphRAG / 图数据库RAG**:主张把代码构造成图结构、做子图/图遍历检索。本项目实现了真正的子图检索(B3)并与文本投影(B0/B1/B2)同口径对照。
- **独立冷静结论**:对**依赖问答**,完整图结构子图检索(B3)**不优于、反劣于**轻量文本化物化(B2)——B3 全体 0.692<B2 0.904(p<0.01),且纯子图丢节点属性致 symbol_location 崩。**价值在"物化所需结构成文本"(给什么),而非"图检索机制多复杂"(怎么取)。** 这给"图RAG对代码理解一定更好"提供一个有数据的边界/反例:至少对依赖问答,轻量物化已足够,完整图RAG是过度工程。

## RLCoder(ICSE2025)深化对比(M24-25 E51)
- **RLCoder**:RL训检索器+perplexity reward,称跨语言泛化超SOTA。本项目实现最小可行版(ppl弱监督)并与静态分析对照。
- **独立结论**:RLCoder在本任务**有效但赢不过静态分析**——ppl弱监督把检索器Recall 0.583→0.958(有效),但<静态分析oracle 1.000;且**ppl-reward 85%指向静态算出的答案**,说明RL收敛向静态分析。与RLCoder文献"学习式检索更优"不一致,口径原因:RLCoder的强项是补全/泛化场景(答案非精确可算),而依赖问答的答案是精确可静态计算的结构事实→静态分析是oracle,学习去逼近它没有超越空间。这给"学习式检索器是否总更优"一个有数据的任务相关边界。
