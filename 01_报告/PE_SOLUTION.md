## 一、四个维度是什么、为什么这么设计

跨文件依赖分析的真 LLM 瓶颈(M3-B 诊断)对应到 PE 的四个抓手:

| PE 维度 | 针对的瓶颈 | 机理 |
|---|---|---|
| System Prompt(角色+格式约束) | 模型答非所问/编造/格式乱 | 约束"只依据上下文、结构化输出、别名先还原" |
| Few-shot 示例库(≥20) | 模型不知道期望的输出格式/推理方式 | 用范例教"列全、还原别名、逐跳追间接依赖" |
| CoT 推理引导 | 间接依赖(多跳)、反向依赖(易漏) | 引导先判类型→定位→逐条核对再下结论 |
| 输出后处理(过滤/排序/去重/补全) | 反向依赖高扇入时模型漏列 | 用检索到的依赖卡片清单补回漏项 |

## 二、System Prompt 模板(项目无关,可直接复用)

```
你是代码库跨文件依赖分析专家。回答依赖类问题时严格遵守:
1. 只依据提供的[依赖图卡片]和代码片段作答,不使用记忆里的旧版本结构,不臆测;
2. 上下文里没有的信息,明确说"提供的信息中未包含",绝不编造文件名/类名;
3. 输出结构化:每条关系写成「文件路径 — 符号名 — 关系类型」;"列出多个"的问题
   用编号列表逐条列全,不要只举几个例子;
4. 遇到别名(import ... as)时,先把别名还原成它指向的真实模块/类,再回答。
```

设计要点:第 2 条压"幻觉"(M3-B 见代码模型把 Response 基类幻觉到 wrappers/__init__.py);
第 3 条压"格式乱 + 只举例不列全";第 4 条直接打别名失联瓶颈。

### 二-bis. humble Prompt(脏依赖场景,可直接复用)
当依赖卡片由静态分析生成、可能漏掉动态/条件/反射依赖时,改用下面这版"承认卡片可能不全"的 System 指令——判分无关下让模型**识破脏依赖的比例大幅提升**(N=103,4 语言)。**口径标注(M76 独立复现)**:**humble 端约 99%(稳健,精确复现);strict 基线对 detect 关键词定义敏感——原 I1 口径约 7%、宽关键词口径约 18-30%**;**跳幅方向在任何合理关键词集下都成立,但"7%"这个具体基线值口径依赖**。原文(可直接复制):

```
你是代码依赖分析助手。以下依赖卡片由[静态分析]生成,可能不完整——运行时动态导入、
条件导入、反射等依赖可能被遗漏或未标注其条件性。请结合卡片与你自己对该项目的了解作答,
对静态分析可能遗漏/误标的依赖要主动指出并说明。
```

适用边界(诚实):humble 让模型**识破/标注**脏依赖的性质(低门槛、真实有效),**但不保证精确补出具体脏机制**(机制定位仅 conditional 3%);且对"下游错卡片是否完整"无判别力(M45)。故 humble 用于"提醒模型卡片可能不全、别盲信",不用于"保证补全"。

## 三、CoT 模板(项目无关)

```
请先一步步分析再给结论:
(1) 判断问题类型:正向依赖(我import谁)/反向依赖(谁import我)/继承/间接多跳/符号定位;
(2) 在卡片与代码里定位相关实体;间接依赖要逐跳追中间节点;别名要先还原;
(3) 逐条核对,把所有命中项列全(尤其反向依赖容易漏);
最后用结构化列表给出结论。
```

## 四、Few-shot 示例库(20 条,按 6 子类型组织)

结构(换项目时按此结构替换示例内容,用被测项目的迷你示例):
- forward_dep(3):教"逐条列出 import 的内部模块,格式 文件—符号—关系";
- reverse_dep(4):教"以卡片的被依赖清单为准**列全**、不够时诚实说不完整";
- inheritance(4):教"跨文件继承标注所在文件、**别名先还原**、多继承列全基类";
- indirect_dep(3):教"逐跳追踪中间节点、区分直接/间接";
- symbol_location/dataflow(4):教"以卡片定位、不靠命名习惯猜";
- 诚实/格式(2):教"没有就说没有、统一输出格式"。

**完整 20 条原文(可直接复制粘贴;示例用通用迷你项目 api/gateway.py、core/models.py、util/log.py 等,不绑定 werkzeug;源文件 `configs/prompts/pe_fewshot.yaml`):**

```
# —— forward_dep ——
Q: 在该项目中,api/gateway.py 直接 import 了哪些项目内模块?
A: 1. core/models.py — Order — 导入数据模型  2. api/auth.py — verify — 导入鉴权  3. util/log.py — log — 导入日志
Q: 某文件 import 了 a.py 和 b.py,如何作答?
A: 逐条列出:1. a.py — 关系:import;2. b.py — 关系:import。只列上下文里确有的。
Q: core/store.py 依赖哪些内部模块?
A: 1. core/models.py — Record — import;若卡片显示无其它内部依赖,则只列这一条。
# —— reverse_dep ——
Q: 哪些文件 import 了 core/models.py?
A: 依据依赖图卡片"被这些文件依赖"一行,逐条列全:1. api/gateway.py 2. core/store.py 3. base/handler.py(卡片里有几个列几个,不遗漏)
Q: 哪些文件依赖 util/log.py?(卡片列出 5 个使用方)
A: 逐条列全 5 个:1. api/gateway.py 2. api/auth.py 3. core/store.py 4. base/handler.py 5. core/models.py
Q: 谁依赖 x.py?但上下文只检索到部分使用方。
A: 列出上下文中确有的使用方,并说明"以上为提供的信息中出现的使用方,可能不完整"。
Q: 反向依赖问题模型容易漏列,怎么避免?
A: 以依赖图卡片的"被依赖"清单为准,逐项抄全,不要凭印象只举几个。
# —— inheritance ——
Q: 类 Child 继承自哪个文件的哪个类?卡片显示"继承自 Parent [base/handler.py]"。
A: Child — 继承自 base/handler.py 的 Parent 类 — 关系:继承(跨文件)。
Q: 类 Foo 的基类写作别名 _Base,如何作答?
A: 先还原别名:_Base 来自 `from core.models import Base as _Base`,即真实基类是 core/models.py 的 Base。Foo — 继承自 core/models.py 的 Base。
Q: 哪些类继承了 Mixin?
A: 依据卡片"继承它的子类"逐条列全:1. A [m1.py] 2. B [m2.py]。
Q: 某类多继承自 X 和 Y,如何作答?
A: 列出全部基类:该类同时继承 X 和 Y 两个类。
# —— indirect_dep ——
Q: A 是否间接依赖 C?路径经过哪个中间文件?
A: 逐跳追踪:A 直接 import B(中间节点),B 直接 import C → A 间接依赖 C,中间文件是 B 的完整路径。
Q: response 经包 __init__ 间接依赖 structures,是直接还是间接?
A: response 直接 import 的是 包/__init__.py(包),structures 是经由包再分发的"间接依赖";中间文件是 包/__init__.py。
Q: 如何判断 X 到 Y 是直接还是间接依赖?
A: 看 X 的 import 段:若直接出现 Y 则直接;若 X 只 import 了 Z 而 Z 再 import Y,则间接,中间节点是 Z。
# —— symbol_location / dataflow ——
Q: 类 Widget 定义在哪个文件?
A: Widget — 定义于 ui/widget.py(依据卡片"定义位置"行)。
Q: 常量 MAX_SIZE 在哪个文件定义?
A: MAX_SIZE — 定义于 core/config.py。
Q: 函数 parse 定义在哪?
A: parse — 定义于 util/parser.py。只给文件,不臆测行号除非卡片提供。
Q: 符号定位题怎么保证准确?
A: 以卡片的"定义位置"或代码块的来源标注为准,不要根据命名习惯猜文件。
# —— 诚实 / 格式 ——
Q: 上下文没有相关信息时怎么回答?
A: 明确回答"提供的信息中未包含该依赖关系",不要编造或用旧版本记忆补全。
Q: 回答依赖问题的统一格式是什么?
A: 每条写「文件路径 — 符号名 — 关系类型」;多项用编号列表;先结论后(可选)简短依据。
```

## 五、输出后处理(过滤/排序/去重/补全 —— 四类各独立量化)

`scripts/postprocess.py` + `scripts/pe27_postproc_quant.py`(M64 判分无关量化)。
考题要求后处理覆盖过滤/排序/去重并**各自独立量化**。下表用**判分无关 keyword_all det-recall + 精度代理**(werkzeug 56 例,gold=测试用例 judge.keywords,零 LLM 裁判)逐类量化:

**可直接复用的后处理规则(`scripts/postprocess.py` 真实逻辑,伪码,他人可照此重实现):**

```
# 0) 仅对"列出多个"类问题做补全(反向/正向依赖、"哪些…"),单答案题(符号定位/间接中间节点)跳过——否则塞噪声
def is_list_q(prompt): return 含("哪些" or "谁依赖" or "列出" or 反向/正向依赖类别)

# 1) 从依赖卡片抽"权威清单项"(只认这几行,避免抓到无关 token)
CARD_LINES = ["它被这些文件依赖/使用", "它依赖(import)的仓库内文件", "继承它的子类"]
def extract_card_lists(prompt):
    items=[]
    for line in prompt.splitlines():
        if any(tag in line for tag in CARD_LINES):
            rhs = line.split(":",1)[-1]                 # 取冒号后清单
            for tok in re.split(r"[,,、\s]+", rhs):
                tok = tok.strip("[]()。.")
                if tok.endswith(".py") or (tok[:1].isupper() and len(tok)>2):  # 文件路径或类名
                    items.append(tok)
    return 去重保序(items)                               # ← 去重(dedup):保序去掉重复项

# 2) 补全(completion):把卡片里有、答案没提到的项追加回去(只用已检索上下文,不引入外部信息)
def postprocess(answer, card_items):
    missing = [it for it in card_items if it not in answer]
    if not missing: return answer
    return answer + "\n[后处理补全 — 依据检索到的依赖卡片清单]: " + ", ".join(missing)

# 3) 过滤(filter,反幻觉接地):剔除"答案里出现但检索上下文完全没有"的文件名 → 本管线 Δ=0(上游已防,见下)
# 4) 排序(sort):非答案级后处理,是检索级——cross-encoder rerank 深度 12→80,把目标卡片排进 top-k
```

| 后处理类 | 做法 | 独立量化(判分无关) | 机理/适用边界 |
|---|---|---|---|
| **排序 sort**(rerank) | cross-encoder 把目标依赖卡片排进 top-k(深度 12→80) | **det/judge +0.14**(`SCALE_STRESS_TEST`/`JUDGE_INDEPENDENT_VALIDATION`) | **四类里最大杠杆**——治"高扇入反向依赖卡片排不进 top-k"(检索瓶颈) |
| **补全 completion** | 从检索卡片抽权威清单,把模型漏列项补回(只用已检索上下文) | **det-recall 0.854→0.901,Δ+0.047**(27/56 用例被补全) | 治"反向依赖漏列"(recall↑);改不了"信息没检索到" |
| **去重 dedup** | 折叠答案里重复列出的同一依赖项 | recall 不变;**精度代理 0.708→0.741,Δ+0.034**(52/56 用例有重复,共去 393 项) | 清洁输出/提精度,不增 recall(不删 distinct 项) |
| **过滤 filter** | 剔除答案里检索上下文完全没有的幻觉文件名(反幻觉接地) | **Δ=0(0/56 用例有幻觉、0 剔除)** | **诚实负结果**:System Prompt 反幻觉约束(§二第 2 条)+ RAG 已在上游杜绝文件名幻觉→接地过滤无可滤、冗余。有判断力的结论:**过滤该放在 System Prompt 层(预防),不是后处理层(事后)** |

**四类小结**:排序(检索深度)是最大杠杆(+0.14);补全帮漏列(+0.047 det-recall);去重提精度(+0.034)不增召回;过滤在本管线 Δ=0(上游 System Prompt 已防幻觉)。**判分无关 det-recall 下补全增益(+0.047)比 §六 LLM-judge 的 +0.01 更大**——因 det-recall 直接量"漏列项补没补回",正是补全的靶心。

### 〔M69 超额〕filter Δ=0 的根因:预防(System Prompt)> 纠正(后处理)
为什么后处理 filter Δ=0?2×2 量化(System 反幻觉约束 on/off × filter,`scripts/pe33_build_sysabl.py`/`pe36_sysabl_analysis.py`):
- **System-OFF**:模型产生**未接地(上下文里没有)的文件名预测 267 个**。
- **System-ON**:同样的题,未接地预测降到 **91 个**——**System Prompt 在源头预防,把编造文件名减 66%**;dep-recall 仅 0.66→0.61(小代价)。
- **filter(后处理纠正)**只能事后剔除这些未接地项;但 **System-ON 预防后已所剩无几,filter 无可纠正(M64 Δ=0)**。
- **工程原则(PE 四维度协同机理,非新发现)**:**同类错误(编造文件名),源头 System Prompt 预防 > 末端 filter 纠正**——预防更彻底、代价更小(不用额外后处理一步)。
- **边界(诚实)**:filter 与 System Prompt 都只对付"编造上下文里没有的文件名"这一类;**对"列了上下文里有但不相关的项"(over-inclusion=precision 问题)两者都不直接管**,那需相关性过滤(另一类后处理)。绝对幻觉率受"路径形 vs 模块形"文本匹配噪声影响,**相对减少 66% 是稳健信号**。

## 六、逐项效果(LLM-judge 为主指标,通用 Qwen2.5-14B + 全栈 RAG)

baseline = 全栈 RAG + 朴素 prompt(LLM-judge 0.77)。每维度独立对照:

| 配置 | 总体 | Δ | 间接依赖 | 继承 | 反向依赖 | 归因 |
|---|---|---|---|---|---|---|
| baseline(plain) | 0.77 | — | 0.40 | 0.89 | 0.69 | — |
| +System | 0.78 | +0.01 | 0.60 | 0.89 | 0.65 | 轻微正;约束减少跑题 |
| +Few-shot(通用20条) | 0.73 | **−0.04** | 0.70 | **0.76** | 0.67 | **负**:通用示例分散注意力、拉低继承 |
| **+CoT** | **0.80** | **+0.03** | **0.70** | 0.89 | 0.68 | **最优单项**:多跳推理 |
| +All(Sys+FS+CoT) | 0.79 | +0.02 | 0.60 | 0.89 | 0.70 | few-shot 拖累,不如纯 CoT |
| +All+后处理 | 0.78 | +0.01 | 0.60 | 0.89 | **0.72** | 后处理只微帮反向依赖漏列 |

**逐项结论与适用边界(对口导师"方案的适用边界"):**
1. **CoT 是最有效的单项(+0.03),增益几乎全部来自间接依赖(0.40→0.70,+0.30)**
   —— 多跳依赖(A→B→C)是**可被思维链引导**的推理瓶颈,CoT 正中靶心。
2. **System Prompt 轻微正(+0.01)**:格式/诚实约束减少跑题,也帮了点间接依赖。
3. **通用 Few-shot 是负结果(−0.04)**:20 条**项目无关**示例反而分散注意力、把继承
   从 0.89 拉到 0.76。诚实记录——**项目无关 few-shot 在有强 RAG 上下文时会与真实
   上下文争夺注意力**。(werkzeug 专属 few-shot 可能更好,但违背泛化要求。)
4. **输出后处理只微帮(+0.01)**:把反向依赖从 0.69 抬到 0.72,但 forward 略降
   (补全偶尔过度)。**后处理改善的是漏列,改不了"信息没检索到"**。
5. **反向依赖高扇入几乎不被任何 PE 改善**(0.69→0.72)—— **这是检索的锅,PE 治不了**。
   它需要的是"把目标依赖卡片稳定排进 top-k",不是更好的 prompt。

**一句话适用边界**:**PE(尤其 CoT)对"推理类"瓶颈(间接/多跳依赖)有效;
对"检索类"瓶颈(高扇入反向依赖,信息不在上下文)无能为力。**

### M6 修订:few-shot 不是"该不该用",而是"用对的"
M4 的"通用 few-shot 有害"结论在 M6 被补全(E26):

| few-shot 类型 | 效果 | 为什么 |
|---|---|---|
| 通用玩具示例(M4) | −0.01 | 与真实 RAG 上下文争注意力 |
| 无 few-shot | 0.80 基线 | — |
| **领域内真实示例(M6)** | **+0.01** | 教对了推理/格式,与任务相关 |

**few-shot 的有效性取决于示例与任务/上下文的相关性。** 精炼后的最优 PE =
**领域 few-shot(≥20,真实依赖 Q&A)+ System + CoT**,配全栈 RAG 达 0.81。

## 七、最终方案 + 瓶颈→解法适用边界总表(项目方法论最终形态)

经六格消融矩阵(E25)+ 两个风格迥异项目验证(werkzeug / rich,E27),最终方案:

> **最优 = 全栈 RAG(依赖图卡片 + 混合检索 + cross-encoder rerank)
> + PE(System + CoT + 领域 few-shot)。微调(FT)出局。**

**瓶颈 → 解法对应表**(哪类瓶颈用哪个手段治,这是整个项目的最终交付):

| 瓶颈 | 性质 | 正确解法 | 数据佐证 |
|---|---|---|---|
| **跨文件关联断裂**(反向依赖:谁依赖我) | 检索/信息缺失 | **RAG**:把依赖图反向边文本化成卡片喂进去 | 反向依赖 0.10(无RAG)→ 0.69(RAG) |
| **别名继承失联**(基类写作 import 别名) | 静态解析 | **静态分析卡片**:解析器还原别名(M3-A) | INH-07 0→1.0(别名卡片) |
| **隐式/多跳依赖**(A→B→C) | 推理 | **PE / CoT**:思维链逐跳追中间节点 | 间接依赖 0.40(plain)→ 0.70(CoT) |
| **高扇入反向依赖**(18 个使用方) | 检索覆盖/长上下文 | **RAG**(把目标卡片排进 top-k);PE/FT 治不了 | 任何 PE 不动它(0.69→0.72);需检索改进 |
| **输出不完整/格式乱** | 生成约束 | **PE System Prompt**(格式约束 + 列全)/ 领域 few-shot | — |

**反过来,什么治不了什么(适用边界):**
- **PE/CoT 治不了检索瓶颈**:信息不在上下文,再好的 prompt 也无中生有不了(反向依赖高扇入)。
- **FT 治不了任何独立瓶颈**:它想抢 PE 的"教格式/推理"的活,但整体 +0.02、与零成本 PE
  分数不可区分(σ≈0.014),且记忆机制泛化弱于 CoT 推理。性价比为负,出局——理由是成本/风险,
  非分数。(注:原稿曾据 5 样本称"FT 把间接砸到 0.20",M8 扩到 n=18 后未复现,已推翻;见 CONFIDENCE.md C9。)
- **RAG 治不了"信息检索到了但要推理串联"**(间接依赖)——那是 CoT 的活。

三手段**各管一段、不可互相替代**:RAG 管检索、PE/CoT 管推理、静态分析卡片管符号失联。

## 八、PE 策略完备性 + 文献背书(M77,不新增提示词实验)

> 结合刚验证的跨模型 PE 边际普适(`CROSS_MODEL_GENERALIZATION.md`),**不再试"魔法提示词让 PE 变好"**——本节确认 PE 覆盖完备性并用 2025 顶会实证为"PE 边际"结论背书。

### 8.1 已系统覆盖的主流 PE 技术
| 技术 | 试过? | 本项目结果 |
|---|---|---|
| zero-shot | ✅ | baseline plain |
| few-shot(≥20 条库) | ✅ | 通用 −0.04 / 领域 +0.01(§四) |
| CoT | ✅ | self-judge 期 +0.03,独立裁判收回为不显著(§六) |
| System / role + 格式 / 反幻觉约束 | ✅ | +0.01,源头防幻觉(§二) |
| humble(承认卡片可能不全) | ✅ | 识破跳幅大(§二-bis) |
| 输出后处理(排序 / 补全 / 去重 / 过滤) | ✅ | 排序 +0.14,余微(§五) |

### 8.2 未试的 PE 技术 + 为何对本任务预期无效(诚实,非"漏了")
**本任务瓶颈是「信息缺失」(跨文件结构不在上下文),非「推理不足」。** 故"优化推理过程"类 PE 预期无效:
| 未试技术 | 它优化什么 | 为何对本任务预期无效 |
|---|---|---|
| self-consistency(自洽) | 多采样推理路径 + 投票 | 结构信息不在上下文时,所有采样都缺它 → 投票不产生新信息 |
| tree-of-thought(ToT) | 搜索推理树 | 同上,优化推理探索、不补检索 |
| least-to-most / 问题分解 | 拆子问题逐步解 | 多跳已由 CoT 处理;不补结构信息 |
| ReAct / prompt-chaining(带检索动作) | 推理 + 检索动作交错 | **本质是"取更多信息" = RAG 家族**;其增益来自补信息(与本项目 RAG 主线一致),非提示模板本身——我们已用 RAG 做了这件事 |

**结论**:**已覆盖主流 PE;未试的"推理优化"类(self-consistency/ToT/分解)因本任务瓶颈是信息缺失而预期无效;"检索交错"类(ReAct)的增益本质属 RAG(已做),非 PE 模板。** 这不是"漏了 X/Y",是"X/Y 优化的维度不是本任务的瓶颈"。

### 8.3 文献背书(MSR 2025)
**Shin et al., 《Prompt Engineering or Fine-Tuning: An Empirical Assessment of LLMs for Code》(MSR 2025)** 实证:**PE 配 GPT-4 不能持续超越微调**(代码生成差 ~28.3pt),真正有效的是**对话式 / 人在回路补充上下文**,而非更好的静态提示模板。
- **与本项目高度一致**:PE 边际、**提升来自补结构信息(RAG)而非优化措辞**。
- **定位升级**:这把"PE 边际"从"我没把提示词写好"**升级为"与顶会实证一致的机理性判断"**——在需要外部结构信息的代码任务上,提升来自**补信息(RAG)**,不来自 PE 模板优化。配合本项目跨模型 PE 边际普适(3 模型双口径),"PE 不是这类任务的杠杆"是机理性结论,非调参不足。
