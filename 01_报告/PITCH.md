## 六段式故事线

### 一、问题:代码依赖关系不在文本里(50 秒)
课题是"跨文件依赖分析"的效果优化。核心难点一句话:**"谁依赖 http.py"这个答案,
根本不在 http.py 自己的代码里**——它分散在另外 18 个文件各自的 import 语句中。
所以这不是"模型不够聪明"的问题,是"信息在哪、怎么喂"的问题。我把整件事先做成
**可测量的**:真实开源项目 werkzeug + 56 条用例(答案由 tree-sitter 静态分析导出、
可核验)+ 一套已校准的评测。

### 二、真实项目诊断:断崖与四类瓶颈(50 秒)
从玩具数据换到真实 werkzeug,baseline 从 0.74+ **断崖到 0.27**——这才是真实难度。
用"检索命中率 × 答案分"交叉定位,把瓶颈精确到机理:**跨文件关联断裂**(反向依赖,
检索到了但答案不在文件文本里)、**别名继承失联**(基类写作 import 别名,物理在场
符号失联)、**多跳间接依赖**、**长上下文丢失**。诊断到机理,才知道每类该用什么药。

### 三、三类瓶颈、三种解法(60 秒)
- **检索的锅 → RAG**:把 tree-sitter 的依赖关系**文本化成"依赖图卡片"**喂进去,
  反向依赖从 0.10 救到 0.69。最强的一条论证:**把原始代码喂给 LLM,分数(0.16)比
  不给上下文(0.18-0.31)还低**——依赖关系不在代码里,必须喂结构化卡片。
- **推理的锅 → PE/CoT**:多跳间接依赖,CoT 思维链逐跳追中间节点,0.40→0.70。
- **符号失联 → 静态分析卡片**:解析器还原 import 别名,别名继承 0→1.0。

### 四、微调:一个诚实的负结果(50 秒)
微调我认真做了:538 条数据、werkzeug 完全留出、QLoRA、过拟合双防线。结果是
**高质量负结果**:FT 整体只 +0.02,分数上**和零成本 PE 不可区分**(0.77 vs 0.80,
差距落在运行方差 σ≈0.014 内)。**微调诚实出局,但出局理由是成本/风险,不是分数。**
——这正是我想讲的第二层:**我连自己的负结果都拿去做了置信审计**。原稿一度宣称
"FT 把间接依赖砸了 −0.20",但那是 5 样本的结论;我把间接用例扩到 18 条复跑,**崩塌
没复现**(ft 0.50 ≈ plain 0.44),于是把这条结论**自己推翻、回填进报告**。敢挤掉自己
结论里的水,比多报一个 0.80 更值钱。

### 五、六格矩阵定论(40 秒)
标准六格(PE/RAG/FT 单项+组合):**RAG 必要且主导**——无 RAG 三格全 ≤0.24,有 RAG
跳到 0.8,这是**远超噪声的铁结论**。但 RAG only 0.77 / PE+RAG 0.80 / All 0.79 三格
**两两差距都小于 2σ,统计上持平**(第二裁判 Coder-14B 下 PE+RAG vs 纯RAG 还会翻转)。
所以我不吹"0.80 比 0.77 强",**只主张"有无 RAG"这一档**;PE 的真实价值在 hard/间接
(n=18:plain 0.44→CoT 0.61,+0.17 才是稳的)。**最优 = RAG + PE(System+CoT)**。
(补充:M11 我把一个欠调的检索深度超参调对后,这套最优方案 0.80→**0.94**、反向依赖 0.70→0.95;
M12 在新深度逐格复查,上面这些相对结论**全保持**——绝对分涨了,故事没变。)

### 六、泛化证明在所测项目上一致成立(30 秒)
换一个**风格迥异、没碰过的**项目 rich(终端渲染库),最优方案把 0.66→0.73,
**提升规律一模一样**(CoT 救间接、RAG 救反向)。所以这是**项目无关的在所测项目上一致成立方案**,
不是对 werkzeug 调出来的特例。最终交付:一张"瓶颈→解法"对应表(FINAL_REPORT 第 5 节)。

## 预想追问(12 个,均指向实验数据)

**Q1:微调没做好,是不是你参数没调对?**
不是,而且我要做两层诚实更正,把这个结论收得更精确。① 训练健康:val loss 0.034→0.0092 单调降不回升,
adapter 也确实加载了(留出的 werkzeug 上 FT only 显著超 baseline 就是证明)——**无 bug**。
② **关键口径核对(M13续)**:我一度说"FT 无增量/性价比为负/伤 hard",但跟文献对照后做了 FT only vs baseline 的直接对比,
发现 **FT 单独其实显著为正**(overall +0.04 / hard +0.11,p<1e-4)——**FT 不是无效,也不伤 hard**。
③ 真相是:FT **弱于 RAG 一个数量级**(+0.04 vs +0.72),在已有 RAG+CoT 的最优配置上**冗余**。所以 FT 不进方案,
**理由是被 RAG 主导且冗余,不是"无效"**——这跟文献(FSE2025 微调正提升但弱于 RAG)完全一致。(E40/E23/E24)

**Q2:怎么证明你没过拟合到测试集?**
三层防线:① **werkzeug 整个留出**,微调一条 werkzeug 数据都没用;② val 监控防背
训练样本;③ **rich(全新项目)泛化复现**了同样的提升规律。要过拟合到 56 条测试集,
这三层都得同时失效。而且我保留了至今全 0 的 hard 用例没去"修",真过拟合早把它们刷绿了。

**Q3:你的方案换个项目还成立吗?**
已验证。rich 是终端渲染库,和 werkzeug 的 web 框架风格完全不同,且没进任何训练。
最优方案在 rich 上 0.66→0.73,**CoT 救间接(0.67→1.00)、RAG 救反向的规律一致复现**
(E27)。选风格不同的项目正是为了排除"同类项目假泛化"。

**Q4:LLM-judge 可信吗?会不会裁判自己也错?**
上线前**认真校准**过:抽 20 条对照 tree-sitter ground truth 人工标注,裁判 vs 人工
**MAE 0.022、95% 一致、0 严重误判**(E20)。我还诚实记录了它的已知弱点(对"拒答+
瞎猜"略给分)。而且校准还**纠正了我自己的错误**——M3 我凭直觉说"keyword 低估
真 LLM",校准发现说反了,偏心在 MockLLM 侧。不校准的裁判我不用。

**Q5:考核要 Few-shot≥20,你却说 few-shot 有害?**
要区分,而且我得带上方差说。**方向上**通用玩具示例偏害、领域真实示例偏益,但 ±0.01
的幅度**落在运行方差 σ≈0.014 内,我不把它当显著结论**。我做了 21 条领域 few-shot 的
对照(E26),真正能站住的是定性原则——**few-shot 有效性取决于示例与任务相关性,
是"用对的"而非"该不该用"**。最优方案含 20+ 条领域 few-shot,作为最佳实践而非刷分手段。

**Q6:RAG 不就是检索代码片段吗,有什么新的?**
关键区别:**我检索的不是代码,是预先算好的结构化依赖知识**。把"谁依赖我/继承自谁/
别名指向谁"用 tree-sitter 算出来文本化成卡片。证据:**直接喂原始代码块,分数比不给
上下文还低(0.16 < 0.18-0.31)**——因为依赖关系不在代码文本里。这条是整个 RAG
路线的立身之本(E18)。

**Q7:为什么用通用 14B 而不是代码专精模型?**
数据说话。纯参数记忆时代码模型确实强(0.31>0.20),但**一旦加 RAG,通用模型每档
反超**(全栈 0.83>0.75,E19)——因为有了依赖卡片,任务从"凭代码理解力回忆"变成
"读懂结构化上下文+follow 指令",这是通用 instruct 模型的主场。公平裁判下结论依然成立。

**Q8:你最大的局限是什么?**
诚实说:**高扇入反向依赖(18 个使用方)是全程唯一没攻克的瓶颈**——任何 PE/FT 都
不动它(0.69→0.72),因为信息装不进检索 top-k。这是**检索覆盖/上下文窗口**问题,
正确方向是提高目标卡片的召回权重或上更大上下文(写进 NEXT_STEPS),不是再堆模型。
能清楚说出"什么治不了什么",本身就是方案成熟的标志。

**Q9:你这些 0.0x 的差距,会不会都是噪声?怎么知道哪个结论可信?**
这正是我 M8 专门做的**置信审计**(`CONFIDENCE.md`),四件事:① 把样本量薄的间接依赖
从 5 扩到 18;② 同配置重跑 3 次标定**运行方差 σ≈0.014**;③ 换 Coder-14B 当**第二裁判**
复核;④ 泛化扩到 pydantic/PyYAML 共 4 项目。结论分三级:**铁结论**(有无 RAG、图卡片
是胜负手——差 0.5+,远超噪声)、**较稳健**(CoT 助间接 +0.17、模型选型——两裁判一致)、
**趋势性**(RAG/PE/FT 三格 0.77~0.80 在噪声内,第二裁判下还翻转 → 我**主动软化成持平**)。
**我宁可把自己的 0.80 说小,也不报一个经不起方差检验的结论。**

**Q10:你说微调没用,但业界/文献明明说微调有效?**
其实我跟文献**不矛盾**,而且我专门做了口径核对(M13续 E40)。① **微调什么**:文献"微调有效"多指微调**检索器/嵌入**
(如 RLCoder)或喂**目标域数据**;我微调的是**生成器**且严守 werkzeug 零参与训练。② **关键:我的结论不是"FT 无效"**——
FT only 显著高于无上下文 baseline(+0.04 overall / +0.11 hard,p<1e-4),**这正是 FSE2025 RAG-or-FT 说的"FT 正提升但弱于 RAG"**。
③ 我精确的结论是:**不喂目标项目数据时,微调生成器正提升但弱于 RAG 一个数量级(+0.04 vs +0.72)、在最优配置上冗余,故不进方案**——
跟文献完全一致。我一度把它说成"性价比为负/伤 hard",那是**口径不精确**(把"强 RAG 之上无边际"说成了"FT 有害"),已诚实更正。(E40)

**Q11:PE 只涨 2%,是不是你 PE 做得太弱?**
不是,有外部参照。① 我做满了四维度(System/CoT/Few-shot/后处理)+ 21 条领域 few-shot,且 CoT 单项已是最有效的;
② **SOTA 实证 PE 在代码任务上的上限就在 ~6% 量级**(arXiv 2602.13890,见 RELATED_WORK)——
我的 CoT 边际 +0.02~0.03 与之**同量级**,不是我弱,是**PE 这条路天花板本就低**;③ 对比之下 RAG 是 +0.5~0.7,
**差一个数量级**——这恰恰是我的核心论点:依赖任务是检索受限型,模型侧优化(含 PE)注定是边际的。
"PE 只涨 2%"不是失败,是**把 PE 的真实上限诚实测出来了**。

**Q12:你怎么发现真瓶颈是检索深度,而不是模型能力?**
靠一个**控制组**。M11 我本来在验证"相似度检索是否被轻视",顺手加了个 deep-mixed 对照(只把 rerank 候选池
从 12 加深到 80,其他不变)。结果它 0.94,反而比我精心设计的融合配置都高——**真杠杆是检索深度,不是模型/融合**。
我又**逐例验证**(REV-01:"谁 import _internal.py"):浅检索没捞到 `_internal.py` 自己的依赖卡片(答案就在卡片的
"被依赖"行里),深检索捞回来了,反向依赖 0.70→0.95。**这个发现是控制变量逼出来的,不是拍脑袋**——
也顺带说明我前 10 天的绝对分被一个欠调超参压着,但相对结论 M12 复查全保持(E37-38)。

**Q13:你的 LLM-judge 会不会是"自己给自己打分"(self-judge)?这不就把结论做虚了吗?**
**会,而且我真栽过——更重要的是我把它揪出来修了,这是我最想讲的一段。** ① 坦白:一位外部评委查出我 werkzeug/rich 的**主结果**
gen 和 judge 都是同一个 Qwen-14B = self-judge,而我 M15 的污染审计还把它标成了"独立裁判"——审计本身有逻辑盲区
(只验"裁判是不是 base",没验"被评是不是同一个 base")。② 我没有辩护,而是回到磁盘逐文件核实(`REBUTTAL_OR_FIX.md`),
确认属实。③ 修复:**生成固定、换三个裁判重判**——self 对照 + 独立 Coder-14B + 独立 internLM2.5-7B(两个不同血统家族),
统计改 56 用例配对。结果:**CoT 那条"小增益"在三个裁判下全部不显著 → 我当场收回**;而 **RAG 主导反被证实更硬**
(deep 0.94 头条独立裁判下 0.915/0.954,p<1e-18,self-judge 反而略低估了 RAG)。④ 还补了**两独立家族 两LLM裁判间一致性 κ=0.806(LLM裁判间,非人工IRR)**,
回应"单裁判无信度"。**结论**:self-judge 只实质污染了 CoT 一条(已删),核心 RAG 结论判分无关。
**这件事我不藏——一个研究的可信度不在于从不犯错,而在于错了能被数据揪出来、并且自己动手修。**(详见 `E44`/`M17_RESULTS.md`)

**Q14:你的测试答案、RAG 卡片、裁判参考都来自同一个 tree-sitter,RAG 碾压是不是"读回卡片"的同义反复?**
**这是评委最尖锐的一刀,我用一个去循环金标准接住了。** ① 我造了一套**与 tree-sitter 完全解耦**的金标准:
用 Python 标准库 `ast`(完全不同的解析器,零 import 项目代码)独立重抽依赖边——归一化后与 tree-sitter 金标准
**forward/reverse/symbol 三类 100% 吻合**(两个独立解析器得出相同答案,本身就证明依赖是源码客观事实、不是 tree-sitter 臆造);
别名继承、传递依赖等难例我**逐行读真实源码**人工核验。② 在这个去循环金标准上,拿同一批答案重判:
**RAG 主效应仍 +0.66~0.70(p<1e-10),和原来的 +0.73~0.77 几乎一样**,"读回卡片"的循环成分至多占 4~15%。
③ 结论:**RAG 数量级主导不是评测循环的伪影,去循环后依然成立。** 而且——提供算出来的依赖结构本来就是 RAG 的作用机制,
去循环检验的是"模型答对的是不是经独立验证的真实依赖",答案是肯定的。(详见 `DECOUPLED_GOLDSET.md`/`E45`)
